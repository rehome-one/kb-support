## 1. Введение

### 1.1. Назначение модуля

`kb-support` — внутренний модуль платформы reHome для приёма, обработки и
сопровождения обращений пользователей (нанимателей, собственников, агентов)
силами операторов поддержки. Модуль реализует функционал helpdesk-системы,
но не как покупное решение, а как часть собственной платформы — в соответствии
с принципом ADR-0001 «разрабатываем сами, минимум внешних сервисов».

Модуль базы знаний (kb-search, kb-wiki) **уже реализован**. kb-support разрабатывается как **отдельный, независимо развёртываемый блок** (свой сервис, своя БД, свой репозиторий), который интегрируется с готовой базой знаний и основной платформой **только по сети (внутренний API)** — без общих таблиц и без включения чужого кода (см. 1.4 и раздел 8).

### 1.2. Почему свой модуль, а не Okdesk

Сравнение helpdesk-сервисов (см. отдельный документ) показало: готовые решения
(Okdesk, HelpDeskEddy, Юздеск) закрывают потребность, но создают зависимость от
внешнего обработчика ПДн и не интегрируются нативно с уже построенной
архитектурой reHome. Собственный модуль:

- работает в едином контуре ФЗ-152 без дополнительного DPA с внешним вендором;
- нативно использует уже существующие сущности (User, Premises, Booking,
  Collaborator) без синхронизации;
- интегрируется с AI-чатом `kb-search` как первой линией поддержки напрямую;
- подчиняется единой системе доступа (Keycloak, access_level, scope из ADR-0003);
- не требует абонентской платы за оператора.

### 1.3. Что воспроизводим из функционала Okdesk

Okdesk ориентирован на выездной сервис и ТОиР оборудования. Мы берём его
**ядро управления заявками** и адаптируем под поддержку пользователей платформы:

| Функция Okdesk | Берём? | Адаптация под reHome |
|---|---|---|
| Учёт и маршрутизация заявок | ✅ | Ядро модуля |
| Контроль SLA | ✅ | По типам обращений и ролям |
| Многоканальный приём заявок | ✅ | AI-чат (главный канал), email, форма, телефон |
| База клиентов / CRM | ✅ частично | Используем существующую модель User, не дублируем |
| Связь заявки с договором | ✅ | Заявка ↔ Booking ↔ Premises |
| Объекты обслуживания / оборудование | ⚠️ переосмыслить | Вместо «оборудования» — квартира (Premises) |
| Чек-листы | ✅ | Для типовых процедур операторов |
| Шаблоны ответов | ✅ | Готовые ответы + интеграция с базой знаний |
| Автоматизация (правила) | ✅ | Триггеры, автоназначение, эскалация |
| Отчёты и аналитика | ✅ | Метрики поддержки |
| Мобильное приложение инженера | ❌ | Не нужно — нет выездных инженеров в ядре |
| Карта, геолокация инженеров | ❌ | Не нужно для поддержки пользователей |
| ТОиР, ППР, складской учёт ТМЦ | ❌ | Вне скоупа |
| Биллинг работ по прайс-листу | ❌ | Платежи — отдельный контур reHome |
| QR-метки оборудования | ❌ | Вне скоупа |

Вывод: мы реализуем **тикет-систему поддержки пользователей**, а не систему
выездного сервиса. Это сознательно меньший и более сфокусированный скоуп.

### 1.4. Связь с существующими модулями

Перечисленные ниже модули **уже реализованы**. kb-support — новый самостоятельный модуль: он **переиспользует их по API**, не включает в свой код и не делит с ними БД. Связь — сетевая, модули деплоятся независимо.

- `kb-search` — AI-чат, **первая линия**. Эскалация уже описана в OpenAPI
  (`POST /chat/sessions/{id}/escalate`). kb-support принимает эти эскалации.
- `kb-auth` — Keycloak, аутентификация операторов и определение scope.
- `kb-staff` — общий staff-кабинет. Рабочее место оператора подключается в него **как отдельный раздел/фронтенд kb-support** через единый вход (kb-auth/Keycloak SSO), без слияния кодовых баз.
- `kb-wiki` / `kb-help` — **готовая** база знаний: kb-support ссылается на статьи и предлагает их (read-only), но **не создаёт и не редактирует** контент БЗ.
- `kb-files` — хранилище вложений (MinIO): используется по API для файлов заявок.
- Основная платформа rehome.one — источник сущностей User, Premises, Booking,
  Collaborator — **только через внутренний API** (см. раздел 4.6).

## 2. Роли и пользователи

### 2.1. Роли в модуле

| Роль | scope (из ADR-0003) | Возможности |
|---|---|---|
| Заявитель | tenant / landlord / agent | Создаёт обращения, видит свои, отвечает, оценивает |
| Оператор 1-й линии | staff_support | Обрабатывает заявки, отвечает, эскалирует |
| Оператор 2-й линии / эксперт | staff_support (расширенный) | Сложные заявки, юридические, финансовые |
| Супервайзер | staff_support + manage | Распределение, контроль SLA, отчёты, управление операторами |
| Администратор | staff_admin | Настройка правил, SLA, шаблонов, каналов |

Роли определяются в Keycloak, scope вычисляется бэкендом из JWT, никогда не
передаётся клиентом (инвариант ADR-0003).

### 2.2. Заявитель — особенность reHome

Заявитель не регистрируется отдельно в kb-support. Это уже существующий
пользователь платформы (наниматель/собственник/агент). Его обращения
автоматически связываются с его профилем, договорами и квартирами.

### 3.2. Статусы заявки (настраиваемый жизненный цикл)

Базовый набор статусов (должен быть конфигурируемым администратором):

```
NEW           → новое, ещё не взято в работу
OPEN          → взято оператором в работу
PENDING       → ожидает ответа заявителя
WAITING       → ожидает третьей стороны (коллаборант, юрист, банк)
ESCALATED     → эскалировано на 2-ю линию
RESOLVED      → решено, ждёт подтверждения заявителя
CLOSED        → закрыто
REOPENED      → переоткрыто после закрытия
```

Переходы между статусами настраиваются (бизнес-процесс). Запрещённые переходы
отклоняются с 422. Каждый переход фиксируется в истории (см. 3.7).

### 3.2.1. Состояние разбирательства (case_state) — для претензионных типов

Базовый `status` отражает работу оператора с тикетом. Для типов COMPENSATION /
GUARANTEE / INSURANCE / ACCEPTANCE_ACT дополнительно ведётся `case_state` —
стадия разбирательства по существу:

```
CLAIM_SUBMITTED → DOCS_PENDING → UNDER_REVIEW → [INSPECTION] →
DECISION_MADE → PAYOUT_PENDING → PAID | REJECTED
```

- Запрещённые переходы отклоняются 422 (как и базовые статусы).
- Каждый переход и решение пишутся в TicketHistory (3.7).
- Переход `PAYOUT_PENDING → PAID` требует подтверждения двумя сотрудниками
  (принцип «4 глаз», flow_TZ_v3 §8) — реализуется правилом AutomationRule,
  а не волей одного оператора.

### 3.3. Типы обращений (специфика reHome)

Настраиваемый справочник. Стартовый набор под предметную область аренды:

```
PAYMENT          → вопросы оплаты, комиссии, сервисного сбора
CONTRACT         → договор, расторжение, условия
MOVE_IN          → заезд, акт приёма-передачи, ключи
MOVE_OUT         → выезд, возврат квартиры
MAINTENANCE      → поломки, ремонт, аварии (связь с коллаборантами)
UTILITIES        → коммунальные платежи
ACCOUNT          → аккаунт, доступ, персональные данные
LISTING          → для собственников: размещение, модерация объявления
COLLABORATOR     → вопросы по услугам коллаборантов
COMPLAINT        → жалоба, спор между сторонами
FRAUD            → подозрение на мошенничество (повышенный приоритет)
OTHER            → прочее
```

### 3.3a. Претензионные типы обращений (flow v3)

В дополнение к базовому справочнику (3.3) вводятся типы, порождаемые сущностями
`reHome_flow_TZ_v3` и Договора найма. Для них ведётся `case_state` (3.2.1) и
заполняются поля 3.1.1.

```
COMPENSATION     → компенсация за ущерб Квартире/имуществу (Договор 5.8)
GUARANTEE        → гарантийная выплата при просрочке + регресс (Договор 5.7)
INSURANCE        → страховой случай по полису партнёра (flow 7.4)
ACCEPTANCE_ACT   → подписание / спор по акту приёма-передачи
```

Маршрутизация по существующим командам (`team` = support / legal / finance);
специализация «claims-оператор» — через scope/тег, без нового значения enum
(см. 8.1).

#### 3.3.1. COMPENSATION — компенсация за ущерб

Самый тяжёлый тип: собственный регламент разбирательства по Договору (5.8.6–5.8.8).

- **Источник / канал.** Заявление наймодателя через ЛК (канал `LK_CLAIM`) или
  эскалация из AI-чата. Привязка к `booking_id` / `premises_id`.
- **Команда.** `legal` + `finance` (специализация claims).
- **Обязательные вложения при приёме (5.8.6):** заявление с описанием
  обстоятельств и суммы; фото/видео ущерба с датой; сравнение с фотофиксацией на
  дату передачи; документы о размере (отчёт об оценке / счёт / прайс); при ущербе
  > 50 000 ₽ — отчёт независимого оценщика. Хранение в kb-files/MinIO (ФЗ-152).
- **Жизненный цикл (`case_state`).** CLAIM_SUBMITTED → DOCS_PENDING →
  UNDER_REVIEW → [INSPECTION] → DECISION_MADE → PAYOUT_PENDING → PAID | REJECTED.
- **Решение оператора (5.8.8):** полная / частичная (обоснованная документально)
  / отказ (с мотивами). Направляется обеим сторонам через ЛК; при выплате —
  перечисление ≤ 10 рабочих дней.
- **Финансовая логика — по ссылке:** зачёт обеспечительного платежа первым
  (5.8.5), Лимит покрытия = 3× месячной платы (5.8.4), перевод через
  `BankProvider.releasePayout`, проводки в `FinancialLedger`, регресс к нанимателю
  (5.8.9). kb-support хранит ссылки, выплату исполняет платёжный контур.
- **Границы.** Оператор не арбитр спора сторон о размере ущерба (5.8.10).
  Перевод денег — принцип «4 глаз» (flow §8) на шаге PAYOUT_PENDING.

#### 3.3.2. GUARANTEE — гарантийная выплата и регресс

Выплата наймодателю при просрочке нанимателя — автоматическая (5.7.7). Тикет
создаётся только когда нужен ручной разбор.

- **Источник / канал.** Системный триггер (`SYSTEM`) при исключениях: признаки
  мошенничества/фиктивности (5.7.7-б), просрочка регресса > 14 дней (5.7.6),
  решение приостановить/прекратить гарантию (5.7.8).
- **Команда.** `finance` (collections).
- **Что делает оператор.** Подтверждает/блокирует авто-выплату при сигналах
  мошенничества; отслеживает регресс (14 дней, плата за рассрочку 0,2 %/день,
  очерёдность зачёта 5.7.6); при неисполнении — действия 5.7.8 (требование,
  взыскание, ограничение доступа).
- **Границы.** Начисление платы за рассрочку и взыскание — платёжный/юридический
  контур; тикет фиксирует решение, не считает деньги сам.

#### 3.3.3. INSURANCE — страховой случай

Оператор сопровождает, но **не является страховщиком**: решение по выплате — за
страховой компанией-партнёром.

- **Источник / канал.** Webhook страхового события (`INSURER_WEBHOOK`, flow 7.4)
  → уведомление в админке; либо сообщение пользователя (`AI_CHAT`).
- **Команда.** `support`.
- **Что делает оператор.** Помогает оформить заявление в страховую, передаёт
  материалы, отслеживает статус у страховщика, информирует пользователя.
- **Отличие от COMPENSATION.** Компенсация — собственное договорное обязательство
  reHome, не страхование (5.8.1). На случай п. 5.8.11 (замена покрытия на
  страхование) тип архитектурно готов перенять поток компенсаций.

#### 3.3.4. ACCEPTANCE_ACT — акт приёма-передачи

До подписания акта обеими сторонами первый платёж лежит на номинальном счёте,
выплата заблокирована. Оператор доводит подписание и фиксирует расхождения
(уточняет общий тип MOVE_IN/MOVE_OUT в части подписания и блокировки денег).

- **Источник / канал.** Одна сторона не подписывает SMS-OTP; расхождения в
  показаниях/состоянии; обращение из чата (`AI_CHAT` / `SYSTEM`).
- **Подтипы.** ACT_MOVE_IN (заезд — блокирует первую выплату наймодателю),
  ACT_MOVE_OUT (выезд — блокирует возврат залога; при ущербе порождает
  COMPENSATION).
- **Команда.** `support` (operations).
- **Что делает оператор.** Напоминания и разъяснения сторонам, переотправка
  SMS-OTP, фиксация расхождений; при ущербе на выезде — создаёт связанный
  COMPENSATION-тикет.
- **Границы.** Оператор не подписывает акт за стороны; разблокировка выплаты —
  следствие подписания обоими, проверяется `PaymentReleaseChecker`.

### 3.4. Каналы поступления

```
AI_CHAT          → эскалация из kb-search (ГЛАВНЫЙ канал)
EMAIL            → входящее письмо
WEB_FORM         → форма на сайте/в приложении
PHONE            → оператор завёл вручную после звонка
INTERNAL         → создано сотрудником
```

На старте обязательны: AI_CHAT, EMAIL, WEB_FORM (по требованию «минимум каналов,
потом расширять»). PHONE и мессенджеры — следующая фаза.

Для претензионных типов добавляются каналы: `LK_CLAIM` (заявление о компенсации
из личного кабинета, 5.8.6), `INSURER_WEBHOOK` (страховое событие от партнёра,
flow 7.4), `SYSTEM` (системный триггер: просрочка платежа, статус акта).

### 3.5. TicketMessage (Сообщение в обращении)

Переписка внутри заявки.

| Поле | Тип | Описание |
|---|---|---|
| `id` | UUID | |
| `ticket_id` | UUID | Ссылка на заявку |
| `author_id` | UUID | Автор (заявитель или оператор) |
| `author_type` | enum | requester / operator / system / ai |
| `body` | text | Текст сообщения |
| `is_internal` | bool | Внутренняя заметка (не видна заявителю) |
| `attachments` | array | Вложения (ссылки на kb-files) |
| `created_at` | datetime | |

Внутренние заметки (`is_internal=true`) видят только операторы — для обсуждения
между собой. Заявитель их не видит. Это важный инвариант доступа.

### 3.6. CannedResponse (Шаблон ответа)

| Поле | Тип | Описание |
|---|---|---|
| `id` | UUID | |
| `title` | string | Название шаблона |
| `body` | text | Текст с переменными ({{requester_name}}, {{ticket_number}}) |
| `type` | enum nullable | К какому типу обращений применим |
| `linked_article_slug` | string nullable | Связанная статья базы знаний |
| `usage_count` | int | Счётчик использований |

Шаблоны интегрированы с базой знаний (kb-wiki): оператор может вставить ответ
со ссылкой на статью help-центра.

### 3.7. TicketHistory (История/аудит)

Неизменяемый журнал всех действий по заявке: смена статуса, переназначение,
изменение приоритета, нарушение SLA. Требование ФЗ-152 и для разбора споров.

| Поле | Тип |
|---|---|
| `id` | UUID |
| `ticket_id` | UUID |
| `actor_id` | UUID |
| `action` | string (status_changed, reassigned, priority_changed, ...) |
| `from_value` / `to_value` | jsonb |
| `created_at` | datetime |

### 3.8. SLAPolicy (Политика SLA)

| Поле | Тип | Описание |
|---|---|---|
| `id` | UUID | |
| `name` | string | Название |
| `applies_to` | jsonb | Условия применения (тип, приоритет, роль заявителя) |
| `first_response_minutes` | int | Норматив первого ответа |
| `resolution_minutes` | int | Норматив решения |
| `business_hours_id` | UUID nullable | График работы (учитывать рабочие часы или 24/7) |
| `priority` | int | Приоритет применения (если подходят несколько) |

Пример: тип=FRAUD → first_response 15 минут, resolution 2 часа.
Тип=LISTING → first_response 4 часа, resolution 1 рабочий день.

Для претензионных типов нормативы выводятся из сроков Договора:

- **COMPENSATION:** окно подачи заявления ≤ 14 кал. дней с обнаружения ущерба
  (валидируется, 5.8.6); рассмотрение ≤ 30 кал. дней с продлением на осмотр/
  оценку и уведомлением (5.8.7); выплата ≤ 10 рабочих дней с даты решения (5.8.8).
- **GUARANTEE:** решение по сигналам — до плановой даты гарантийной выплаты
  (5.7.2); контроль регресса 14 кал. дней, плата за рассрочку 0,2 %/день
  (5.7.5–5.7.6).
- **INSURANCE:** первый ответ пользователю — по базовому SLA; срок страховщика
  отслеживается как внешний.
- **ACCEPTANCE_ACT:** каденция напоминаний; эскалация при неподписании в N дней
  (блокирует выплату).

### 3.9. AutomationRule (Правило автоматизации)

Аналог автоматизаций Okdesk. Триггер → условие → действие.

| Поле | Тип | Описание |
|---|---|---|
| `id` | UUID | |
| `name` | string | |
| `trigger` | enum | on_create / on_update / on_sla_breach / time_based |
| `conditions` | jsonb | Условия (тип, приоритет, канал, ключевые слова) |
| `actions` | jsonb | Действия (назначить, сменить статус, тег, уведомить, эскалировать) |
| `is_active` | bool | |
| `order` | int | Порядок применения |

Примеры правил:
- Если тип=FRAUD → приоритет=critical, назначить на команду legal, уведомить супервайзера.
- Если канал=AI_CHAT и нет ответа оператора 30 мин → эскалировать.
- Если заявитель не отвечает в статусе PENDING 3 дня → автозакрытие с уведомлением.

### 3.10. Что НЕ дублируем

Сущности User, Premises, Booking, Collaborator **уже существуют** в платформе.
kb-support ссылается на них по id, не копирует. CRM-функциональность Okdesk
(база клиентов, договоров) у нас уже реализована в основной платформе —
переиспользуем, не пишем заново. Это прямое следствие ADR-0001.


### 3.11. TicketCaseDetails (детали претензионного обращения)

Связанная с Ticket сущность 1:1 для типов COMPENSATION / GUARANTEE / INSURANCE /
ACCEPTANCE_ACT. Хранит тип-специфичный payload, чтобы не раздувать Ticket
десятками nullable-колонок.

| Поле | Тип | Описание |
|---|---|---|
| `id` | UUID | |
| `ticket_id` | UUID | Ссылка на заявку (1:1) |
| `case_type` | enum | COMPENSATION / GUARANTEE / INSURANCE / ACCEPTANCE_ACT |
| `payload` | jsonb | Структура по типу (валидируется схемой) |
| `created_at` / `updated_at` | datetime | |

Примеры payload по типам:

- **COMPENSATION:** `limit_remaining`, `deposit_offset_amount`, `evidence[]`,
  `independent_appraisal` (bool), `late_submission` (bool).
- **GUARANTEE:** `missed_payment_id`, `guarantee_payout_id`, `regress_due_at`,
  `late_fee_accrued`, `guarantee_paused`.
- **INSURANCE:** `insurer_claim_ref`, `insurer_status`, `event_payload`.
- **ACCEPTANCE_ACT:** `act_kind` (MOVE_IN/MOVE_OUT), `signing_status`
  (one_signed/both_signed/disputed), `blocked_payment_id`.

Вложения-доказательства (фото/видео/оценка) — в kb-files/MinIO, доступ по
`access_level` (ФЗ-152).

## 4. Функциональные требования

### 4.1. Приём заявок

- FR-1.1. Создание заявки из эскалации AI-чата: `kb-search` вызывает API
  kb-support, передаёт историю диалога, профиль пользователя, контекст
  (premises_id/booking_id если есть). Создаётся Ticket с channel=AI_CHAT,
  chat_session_id заполнен.
- FR-1.2. Создание заявки из email: входящие письма на support@rehome.one
  парсятся, создают/дополняют заявку. Ответы по email привязываются к заявке
  по номеру в теме.
- FR-1.3. Создание заявки через веб-форму в личном кабинете.
- FR-1.4. Создание заявки оператором вручную (channel=PHONE/INTERNAL).
- FR-1.5. Автоматическое связывание заявки с профилем заявителя, его активным
  договором и квартирой (если однозначно определяется).

### 4.2. Обработка заявок оператором

- FR-2.1. Рабочее место оператора: список заявок с фильтрами (статус, тип,
  приоритет, назначенный, SLA-дедлайн), сортировкой, цветовой индикацией
  приоритета и нарушений SLA.
- FR-2.2. Карточка заявки: вся переписка, контекст заявителя (профиль, договор,
  квартира, история прошлых обращений), внутренние заметки, действия.
- FR-2.3. Ответ заявителю (через тот же канал, откуда пришла заявка).
- FR-2.4. Внутренние заметки между операторами (не видны заявителю).
- FR-2.5. Вставка шаблона ответа со ссылкой на статью базы знаний.
- FR-2.6. Смена статуса, приоритета, типа, тегов.
- FR-2.7. Назначение/переназначение заявки на оператора или команду.
- FR-2.8. Эскалация на 2-ю линию.
- FR-2.9. Связывание заявки с коллаборантом (если вопрос про услугу — например,
  заявка на ремонт направляется через ServiceOrder).

### 4.3. Маршрутизация и автоназначение

- FR-3.1. Автоматическое назначение по правилам (AutomationRule): по типу,
  каналу, ключевым словам, загрузке операторов (round-robin / по наименьшей
  загрузке).
- FR-3.2. Распределение по командам (support / legal / finance).
- FR-3.3. Ручное переназначение супервайзером.

### 4.4. SLA

- FR-4.1. Применение SLA-политики при создании заявки по условиям.
- FR-4.2. Расчёт дедлайнов первого ответа и решения с учётом графика работы
  (рабочие часы или 24/7).
- FR-4.3. Индикация приближения и нарушения дедлайна (цветовая, уведомления).
- FR-4.4. Эскалация при нарушении SLA (через AutomationRule).
- FR-4.5. Учёт пауз: время в статусе PENDING (ожидание заявителя) не
  засчитывается в SLA решения.

### 4.5. База знаний и шаблоны

> kb-support интегрируется с **уже готовой** базой знаний (kb-wiki/kb-search) в режиме read-only: ищет и предлагает статьи, ссылается на них из шаблонов. Создание и ведение контента БЗ — вне этого модуля.

- FR-5.1. Управление шаблонами ответов (CRUD, scope=staff_support+).
- FR-5.2. Подстановка переменных в шаблон.
- FR-5.3. Связь шаблона со статьёй kb-wiki — при ответе предлагать релевантную
  статью.
- FR-5.4. Предложение статей базы знаний по содержанию заявки (используя
  существующий поиск kb-search).

### 4.6. Интеграция с платформой

- FR-6.1. Получение данных заявителя (User), договора (Booking), квартиры
  (Premises), коллаборанта (Collaborator) из основной платформы.
  Способ интеграции — **через внутренний API rehome.one** (решено в ADR-0005,
  см. раздел 8): HTTP-клиент с кешированием в Redis, без прямых SQL к чужим БД.
- FR-6.2. Приём эскалаций от kb-search через внутренний API.
- FR-6.3. Возврат ответа оператора в AI-чат: когда оператор отвечает по заявке
  с channel=AI_CHAT, ответ доставляется пользователю обратно в чат-сессию.
- FR-6.4. Создание ServiceOrder для коллаборанта из заявки типа MAINTENANCE.
- FR-6.5. Уведомления заявителю (email, push, в чат) о новых ответах и смене
  статуса.

### 4.7. Аналитика и отчёты

- FR-7.1. Панель супервайзера: открытые заявки, нарушения SLA, загрузка
  операторов, среднее время ответа/решения.
- FR-7.2. Отчёты: объём заявок по типам/каналам/периодам, соблюдение SLA,
  удовлетворённость (рейтинги), повторные обращения, эффективность операторов.
- FR-7.3. Метрики для дашборда Grafana (см. инфраструктурный how-to): rate
  заявок, containment rate AI-чата (доля без эскалации), время в очереди.

### 4.8. Оценка качества

- FR-8.1. После закрытия заявки заявителю предлагается оценить (1-5) и
  оставить комментарий.
- FR-8.2. Низкие оценки (1-2) автоматически уведомляют супервайзера.


### 4.9. Обработка претензионных обращений (flow v3)

- FR-9.1. Приём заявления о компенсации (COMPENSATION) с обязательными вложениями
  (5.8.6); валидация комплекта; при сумме > 50 000 ₽ — требование отчёта
  независимого оценщика.
- FR-9.2. Ведение `case_state` (3.2.1); запрос документов (DOCS_PENDING) и
  инициирование осмотра/оценки (INSPECTION).
- FR-9.3. Оформление решения: FULL / PARTIAL / REJECTED с обязательной
  мотивировкой для частичной выплаты и отказа; уведомление обеих сторон через
  ЛК (5.8.8).
- FR-9.4. Выплата по решению — через API платёжного контура (BankProvider), с
  зачётом обеспечительного платежа первым (5.8.5) и контролем Лимита покрытия 3×
  месячной платы (5.8.4); подтверждение «4 глаз» на шаге PAYOUT_PENDING.
- FR-9.5. GUARANTEE: тикет создаётся системным триггером только при исключениях
  (мошенничество, просрочка регресса > 14 дней, приостановка гарантии);
  отслеживание регрессного обязательства и платы за рассрочку.
- FR-9.6. INSURANCE: приём страхового события (webhook партнёра), сопровождение
  пользователя и передача в страховую; решение по выплате — за страховщиком.
- FR-9.7. ACCEPTANCE_ACT: доведение двустороннего подписания (напоминания,
  переотправка SMS-OTP), фиксация расхождений; при ущербе на выезде — создание
  связанного COMPENSATION-тикета.
- FR-9.8. Все денежные операции (выплаты, регресс, плата за рассрочку, проводки
  FinancialLedger) исполняются платёжным контуром; kb-support хранит решение и
  ссылки, не считает деньги сам.

## 5. Нефункциональные требования

### 5.1. Безопасность и ФЗ-152

- NFR-1.1. Все данные — в РФ, шифрование в покое и в передаче.
- NFR-1.2. access_level на каждой заявке. Заявитель видит только свои заявки.
  Оператор — по своей команде/scope. Фильтрация на уровне хранилища (ADR-0003).
- NFR-1.3. Внутренние заметки (is_internal) недоступны заявителю — критичный
  инвариант, покрыть security-тестом.
- NFR-1.4. Полный аудит действий (TicketHistory) — хранение 5 лет.
- NFR-1.5. Маскирование ПДн в логах и при передаче в AI-чат.
- NFR-1.6. scope из JWT, не от клиента.

### 5.2. Производительность

- NFR-2.1. Список заявок оператора — p95 < 500 мс при 10 000 активных заявок.
- NFR-2.2. Создание заявки из эскалации — p95 < 1 с.
- NFR-2.3. Cursor-based пагинация для списков (как в основном API).

### 5.3. Надёжность

- NFR-3.1. Не терять заявки: приём идемпотентен (Idempotency-Key), при сбое
  внешнего канала — ретраи.
- NFR-3.2. SLA-таймеры устойчивы к перезапуску сервиса (хранятся в БД, не в памяти).

### 5.4. Соответствие стилю проекта

- NFR-4.1. REST + OpenAPI 3.1, как основной API (ADR из проекта).
- NFR-4.2. Стек: Django/FastAPI, PostgreSQL, Redis, Dramatiq (для SLA-таймеров
  и фоновых правил), как в ADR-0001.
- NFR-4.3. Двухконтурность, cursor-пагинация, RFC 7807 для ошибок — как в
  основном API базы знаний.
- NFR-4.4. **Независимое развёртывание:** kb-support — отдельный сервис со своей
  БД и репозиторием; связь с базой знаний и платформой — только по сети, без общих
  таблиц. Сбой kb-support не влияет на работу базы знаний и наоборот.

## 6. API (контур)

Модуль предоставляет REST API, описанный в OpenAPI 3.1. Группы эндпоинтов:

```
# Заявки
GET    /api/v1/support/tickets                 список (фильтры, пагинация)
POST   /api/v1/support/tickets                 создать
GET    /api/v1/support/tickets/{id}            карточка
PATCH  /api/v1/support/tickets/{id}            обновить (статус, приоритет, ...)
POST   /api/v1/support/tickets/{id}/assign     назначить
POST   /api/v1/support/tickets/{id}/escalate   эскалировать
POST   /api/v1/support/tickets/{id}/resolve    решить
POST   /api/v1/support/tickets/{id}/close      закрыть
POST   /api/v1/support/tickets/{id}/reopen     переоткрыть
POST   /api/v1/support/tickets/{id}/rate       оценка заявителя
POST   /api/v1/support/tickets/{id}/decision   решение по претензии (full/partial/reject)
POST   /api/v1/support/tickets/{id}/case-state переход состояния разбирательства

# Сообщения
GET    /api/v1/support/tickets/{id}/messages   переписка
POST   /api/v1/support/tickets/{id}/messages   добавить сообщение/заметку

# Интеграция с AI-чатом
POST   /api/v1/support/tickets/from-chat       создать из эскалации kb-search

# Шаблоны
GET    /api/v1/support/canned-responses        список
POST   /api/v1/support/canned-responses        создать
PATCH  /api/v1/support/canned-responses/{id}   изменить

# SLA и правила (admin)
GET    /api/v1/support/sla-policies            список
POST   /api/v1/support/sla-policies            создать
GET    /api/v1/support/automation-rules        список
POST   /api/v1/support/automation-rules        создать

# Аналитика
GET    /api/v1/support/stats                   метрики
GET    /api/v1/support/reports/{type}          отчёты

# Webhooks (для возврата ответа в чат, уведомлений)
POST   /api/v1/support/webhooks                регистрация
```

Машиночитаемый контракт (**OpenAPI 3.1 v1.1**) приведён в **Приложении A** и
сопровождается отдельным файлом `reHome_kb_support_openapi_v1.1.yaml`. Контракт
уже включает претензионные типы: значения enum `type`
(COMPENSATION/GUARANTEE/INSURANCE/ACCEPTANCE_ACT) и `channel`
(LK_CLAIM/INSURER_WEBHOOK/SYSTEM), поля 3.1.1, схемы `TicketCaseState`,
`TicketDecision`, `ActKind`, `SigningStatus`, `TicketCaseDetails`, действия
`/decision` и `/case-state`, события webhook `case_decided`, `payout_released`,
`insurance_event`. Контракт фиксируется на E1 и расширяется только через PR без
слома обратной совместимости.

## 8. Принятые архитектурные решения

Решения по ключевым развилкам приняты и зафиксированы в **ADR-0005 «Модуль
службы поддержки»**. Кратко:

1. **Интеграция с платформой** (FR-6.1): **через внутренний API rehome.one**
   (слабая связанность, независимый деплой). Доступ к данным User / Booking /
   Premises / Collaborator — через HTTP-клиент с кешированием в Redis, не
   прямыми SQL-запросами к чужим таблицам.
2. **Граница с ServiceOrder**: **разные сущности, связанные по ссылке**. Ticket
   — общение с жильцом; ServiceOrder (ADR-0004) — работа коллаборанта с эскроу.
   Заявка MAINTENANCE может породить ServiceOrder, в Ticket сохраняется
   `service_order_id`, но жизненные циклы независимы.
3. **Email-канал**: **свой IMAP-парсер в контуре РФ** (ФЗ-152, без внешнего
   обработчика ПДн). Реализуется фоновым воркером на Dramatiq.
4. **Уведомления заявителю**: **email + push + возврат в AI-чат + SMS через
   Exolve**. Единый сервис уведомлений с роутингом по каналам и дедупликацией
   (защита от спама — на этапе реализации).
5. **Объём UI 1-й версии**: **минимум — список заявок с фильтрами + карточка
   заявки**. Полноценное рабочее место с дашбордами — следующая фаза, по мере
   роста команды (на старте 1-3 оператора).

Подробное обоснование, альтернативы и технические следствия — в ADR-0005.

### 8.1. Решения и открытые вопросы по претензионным типам (flow v3)

Принятые принципы:

- Решение по компенсации принимается и хранится в kb-support (это работа
  оператора), фактическая выплата — по API платёжного контура (согласуется с
  Решением 1).
- `TicketCaseDetails` — jsonb-payload + схема-валидатор на тип (гибкость для MVP).
- «4 глаз» дублируется: подтверждение в kb-support (переход `case_state`) +
  проверка на стороне выплаты (`releasePayout`).

Открытые вопросы (для Архитектора):

1. Окно подачи 14 дней (5.8.6): жёсткий отказ при просрочке или приём с флагом
   `late_submission` и решением оператора? Рекомендация — приём с флагом.
2. Где начисляется плата за рассрочку 0,2 %/день (5.7.5): в платёжном контуре или
   в kb-support по триггеру? Рекомендация — платёжный контур, тикет отображает.
3. Нужен ли отдельный scope/роль «claims-оператор» поверх staff_support, или
   достаточно тегов и команд legal/finance? Рекомендация — тег + команда на
   старте, отдельный scope при росте.

> **Апстрим-зависимости (flow_TZ_v3 §9, ещё не реализованы):** `AcceptanceAct`
> (двустороннее SMS-OTP), `PaymentReleaseChecker`, `FinancialLedger`,
> `BankProvider.releasePayout`, обеспечительный платёж, сущность регрессного
> обязательства, webhook страховщика. Пока их нет — E10 реализуется в режиме
> «фиксация решения + ссылки», фактическая выплата подключается по готовности.

## 9. Что осознанно НЕ делаем (антискоуп)

Чтобы не раздуть модуль, явно фиксируем, чего НЕ реализуем (в отличие от Okdesk):

- Выездное обслуживание, мобильное приложение инженера, геолокация, карта.
- ТОиР, планово-профилактические работы (ППР), графики обслуживания оборудования.
- Складской учёт ТМЦ, запчасти, списания.
- Биллинг работ по прайс-листу (платежи — отдельный контур reHome).
- QR-метки оборудования.
- Полноценная CRM (используем существующую модель платформы).
- Телефония как встроенный модуль (на старте — оператор заводит заявку вручную
  после звонка; интеграция телефонии — отдельная задача при росте).
- Арбитраж спора сторон о размере ущерба: оператор оформляет решение reHome о
  компенсации, но не разрешает спор между нанимателем и наймодателем (Договор
  5.8.10) и не считает деньги — это платёжный/юридический контур.

Эти функции либо не нужны для поддержки пользователей платформы аренды, либо
уже реализованы в других модулях reHome.
