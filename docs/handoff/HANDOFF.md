# HANDOFF — kb-support

> Передаточный пакет для запуска разработки модуля службы поддержки reHome.
> Bootstrap-дата: 2026-05-30. Архитектор: Evgeniy.

## TL;DR

`kb-support` — отдельный (независимо развёртываемый) модуль платформы reHome
для приёма, обработки и сопровождения обращений пользователей. Реализует
ядро helpdesk-системы класса Okdesk, адаптированное под аренду жилья. ТЗ
описывает 10 этапов реализации (E1-E10) на ~16-17 недель.

**Архитектурный инвариант:** kb-support — независимый сервис, своя БД, свой
репозиторий. Связь с уже реализованным модулем базы знаний (rehome-kb-platform)
и с основной платформой rehome.one — **только по сети, по API**. Без общих
таблиц и без общего кода.

## Что в этой папке

- `04_openapi.yaml` — **машиночитаемый контракт API** (OpenAPI 3.1 v1.1).
  Извлечён из Приложения A `01_TZ_kb_support_v2.2.md` для использования
  в mock-серверах / кодогенерации / контрактных тестах.
  **Известный TD:** spec содержит legacy `nullable: true` (deprecated в
  OpenAPI 3.1); strict Redocly lint падает с 28 ошибками. Handoff yaml

  (см. `state-of-code.md` TD-OAS-001).
- `04_openapi_v1_outdated.yaml` — **archival** более ранняя v1 (без
  претензионных типов). Не использовать — оставлено для исторической
  трассировки.

## Что НЕ в этой папке (внешние зависимости)

### Из rehome-kb-platform (kb-search / kb-wiki / kb-files / kb-auth)

Уже реализованы и эксплуатируются. kb-support использует их **по API**:

- **kb-search** — AI-чат, **первая линия** поддержки. Эскалация уже описана
  в OpenAPI существующего модуля (`POST /chat/sessions/{id}/escalate`).
  kb-support принимает эти эскалации (FR-1.1 ТЗ).
- **kb-auth** — Keycloak SSO, аутентификация операторов и определение scope.
- **kb-wiki / kb-help** — база знаний для предложения статей оператору
  и подстановки в шаблоны (read-only через API).
- **kb-files** — хранилище вложений (MinIO) для файлов заявок.

### Из основной платформы rehome.one

Источник сущностей User, Premises, Booking, Collaborator, ServiceOrder —
**только через внутренний API** (ADR-0005 Решение 1). Без прямых SQL-запросов
к чужим таблицам.

### Договор найма + flow_TZ_v3 (нужны для E10)

Источник сроков и процедур для претензионных типов (COMPENSATION / GUARANTEE /
INSURANCE / ACCEPTANCE_ACT). На bootstrap'е этих документов в репозитории нет;
ТЗ ссылается на них по разделам (5.7 / 5.8 Договора; §7.4 / §8 / §9 flow_TZ_v3).
Архитектор предоставит исходники при подходе к E10 при необходимости.

**Upstream-зависимости (отмечено в §8.1 ТЗ, ещё не реализованы в платформе):**
`AcceptanceAct`, `PaymentReleaseChecker`, `FinancialLedger`,
`BankProvider.releasePayout`, обеспечительный платёж, сущность регрессного
обязательства, webhook страховщика. E10 реализуется в режиме «фиксация
решения + ссылки», фактическая выплата подключается по готовности этих
upstream-модулей.

## Принятые архитектурные решения (на момент bootstrap'а)

- **ADR-0005** — Модуль службы поддержки (5 решений: интеграция через API,
  граница с ServiceOrder, свой IMAP-парсер в РФ, каналы уведомлений,
  минимальный UI на старте). См. `../adr/0005-support-module.md`.
- **ADR-0001 / 0003 / 0004** — наследуются из kb-platform (стек, двухконтурность,
  модель коллаборантов). Копии в `../adr/` как reference-снимок на момент
  bootstrap'а.

## Открытые вопросы Архитектору (для E10)

Зафиксировано в §8.1 ТЗ, требуется решение **до начала E10**:

1. **Окно подачи 14 дней (Договор 5.8.6):** жёсткий отказ при просрочке
   vs приём с флагом `late_submission` + решение оператора?
   Рекомендация ТЗ — приём с флагом.
2. **Плата за рассрочку 0.2 %/день (Договор 5.7.5):** где начисляется —
   в платёжном контуре или в kb-support по триггеру?
   Рекомендация ТЗ — платёжный контур, тикет отображает.
3. **Scope «claims-оператор»:** отдельный scope поверх staff_support
   vs тег + команда `legal`/`finance`?
   Рекомендация ТЗ — тег + команда на старте, отдельный scope при росте.

## Definition of Done (модуль в целом)

См. раздел 7 главного ТЗ. Резюме:
- OpenAPI спецификация полная, проходит линтер.
- Все эндпоинты покрыты тестами (≥80% бизнес-логика).
- Security-тесты: заявитель не видит чужие заявки и внутренние заметки.
- SLA-таймеры переживают перезапуск сервиса.
- Интеграция с kb-search работает end-to-end (эскалация → ответ в чат).
- Двухконтурность и access_level соблюдены, проверено тестами.
- Метрики выведены в Grafana.
- ФЗ-152 чек-лист подписан.
- Документация для операторов написана.
