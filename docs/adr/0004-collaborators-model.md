# ADR-0004: Единая модель коллаборантов

## Статус

- [x] **Принято**
- **Дата:** 2026-05
- **Автор:** Архитектор проекта
- **Согласовано Архитектором:** да

## Контекст

Платформа reHome работает со многими внешними организациями:
- УК, ТСЖ — для решения вопросов общедомового имущества
- Аварийные службы (вода, газ, электричество)
- Клининговые компании (Qlean, Чистофф)
- Переездчики (Газелькин, Деловые Линии)
- Мелкий ремонт (Ленремонт, Profi.ru)
- Доставка ключей (Яндекс Доставка)
- Страховые
- Банк-партнёр, KYC-провайдер, SMS, ЭДО, КЭП

Изначально это были два параллельных мира:
- «Партнёры» — категория C технологического стека (бизнес-договоры)
- «Контакты УК/ТСЖ» — в карточке квартиры (просто данные для жильца)

Это создавало проблему: один и тот же реальный субъект (УК, страховая)
заводится в двух местах с разной семантикой. Нет единой точки учёта,
аналитики, договоров.

## Решение

Принята единая сущность **Collaborator** с подтипами и разными финансовыми
моделями.

### 14 типов коллаборантов

```
- management_company   — ТСЖ, ЖСК, УК
- emergency_service    — аварийные службы (вода, газ, электричество, лифты)
- repair_handyman      — мастера на час (сантехники, электрики)
- cleaning             — клининговые компании
- moving               — переездчики
- key_delivery         — доставка ключей
- insurance            — страховые
- payment_partner      — банк-партнёр (Точка, Т-банк)
- kyc_provider         — Cyclops, Dadata, Контур.Фокус
- edo_provider         — Контур.Диадок, HRlink
- sms_voice            — Exolve, МТС-ID
- it_infrastructure    — Yandex Cloud, Selectel
- legal_consultant     — юр. фирмы, нотариусы
- other                — индивидуально
```

### 4 финансовые группы

Утверждённое закрепление (изменение типа → группы требует ADR):

| Группа | Кто кому платит | Типы коллаборантов |
|---|---|---|
| **A** | Платформа → коллаборанту | payment_partner, kyc_provider, sms_voice, it_infrastructure, edo_provider, legal_consultant |
| **B** | Пользователь → платформа → коллаборант (с комиссией 10-25%) | cleaning, moving, key_delivery, repair_handyman |
| **C** | Реферальная — пользователь платит коллаборанту, мы получаем bonus | insurance |
| **D** | Бесплатные контакты, без денежных отношений | management_company, emergency_service |

### 3 уровня портала (portal_access_level)

Выбирает сам коллаборант при онбординге:
- **NONE** — без личного кабинета (records-only). По умолчанию для группы D.
- **LIGHT** — push/email + список заявок + история выплат
- **FULL** — дашборд + аналитика + управление расписанием + ответы на отзывы

Понижение — в любой момент. Повышение — после первой успешной операции
или одобрения оператора.

### Привязка к объектам

Связь N:N через таблицу `PremisesCollaborator`:
- УК — привязана к ДОМУ (auto-fill для всех квартир в одном доме)
- Аварийная служба — к району
- Конкретный мастер — к квартире
- Поля: role (default_uk, emergency_water, plumber и т.д.), priority

### Заказы услуг

Для группы B вводится сущность `ServiceOrder`:
- Пользователь создаёт заказ → деньги в HOLD (эквайринг)
- Коллаборант получает webhook → ACCEPT/REJECT
- После COMPLETED — выплата за вычетом комиссии
- При CANCELLED — возврат по правилам коллаборанта

## Альтернативы

1. **Два параллельных мира («партнёры» + «контакты УК»)** — отклонено:
   удвоение модели данных, рассинхронизация, нет единой аналитики.

2. **Только справочник контактов без бизнес-логики** — отклонено: тогда
   невозможна монетизация через клининг/ремонт, нет договоров с партнёрами.

3. **Marketplace-модель** (как Profi.ru) — отклонено для текущей фазы:
   избыточная сложность, не основной бизнес reHome (аренда).

## Последствия

### Положительные

- Одна сущность вместо двух — проще для команды, для UI, для отчётности
- Универсальный CRUD: одна форма заведения, одна карточка
- Единая аналитика: сравнимые метрики между типами коллаборантов
- Возможность монетизации через клининг/ремонт (группа B) с эскроу
- Чёткое разделение финансовых моделей — не путать в учёте
- Само-онбординг коллаборантов через rehome.one/partners снимает нагрузку
  с операторов

### Отрицательные / компромиссы

- Сложная модель для команды (14 типов × 4 группы × 3 уровня кабинета)
- Различия в поведении системы для разных типов — может запутывать
- Финансовая логика B-группы (эскроу, комиссия, выплаты) — отдельная
  ответственность, требует контрактных тестов с банком
- Юридические договоры (для B-группы) — комиссионное соглашение, требует
  юриста для составления шаблона

### Технические следствия

- Новые модели данных: `Collaborator`, `PremisesCollaborator`,
  `CollaboratorReview`, `ServiceOrder`, `CollaboratorAccessHistory`
- 11 новых эндпоинтов в API (см. ПЗ API раздел 3.10)
- 14 новых типов webhook events (см. OpenAPI WebhookEvent enum)
- UI для онбординга на rehome.one/partners — публичная форма
- UI личного кабинета коллаборанта (3 уровня) — отдельный модуль
- Auto-fill УК по адресам через Dadata + открытые реестры
- Эскроу-логика для ServiceOrder — расширение платёжного контура
