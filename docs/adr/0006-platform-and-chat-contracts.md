# ADR-0006: Провизорные контракты rehome.one platform API и возврата ответа в kb-search (для E3)

## Статус

- [x] **Предложено** (черновик на ревизию Архитектора)
- [ ] Принято
- **Дата:** 2026-06-01
- **Автор:** Агент-Разработчик (черновик)
- **Согласовано Архитектором:** нет — ожидает подписи

> **ВАЖНО.** Этот ADR фиксирует **провизорные** (временные, subject to change)
> сетевые контракты. Реальные интерфейсы upstream (`rehome.one` platform и
> входящий приём ответа на стороне `kb-search`) ещё **не зафиксированы** — это
> блокер **TD-PA-001** (`docs/state-of-code.md`). Контракт ниже — наша рабочая
> гипотеза, против которой E3 строит клиенты **за adapter-интерфейсом**, чтобы
> смена upstream не текла в доменный код. При фиксации реального контракта этот
> ADR обновляется (или заменяется), меняются только адаптеры.

## Контекст

Этап **E3** (эпик #16) — двусторонняя интеграция с AI-чатом `kb-search` и
получение контекста заявителя из основной платформы `rehome.one`:

- **Контекст заявителя** (#71, E3-3): оператору на карточке нужны данные
  User / Premises / Booking / Collaborator. По **ADR-0005 Решение 1** доступ —
  только по HTTP к platform API с кешированием, без общей БД.
- **Возврат ответа** (#72, E3-4): ответ оператора (не внутренняя заметка) должен
  вернуться в исходную chat-session пользователя в `kb-search`.

Проблема: интерфейсы обеих сторон upstream не зафиксированы (TD-PA-001). Решение
Архитектора (2026-06-01): **не ждать**, а зафиксировать провизорный контракт в
этом ADR и строить клиенты за adapter'ом против него (разблокирует #71/#72).
Якорь доверия для m2m — общий Keycloak (как в #29): `kb-support` ходит к соседям
с Bearer JWT (Client Credentials).

## Решение

### Решение 1. Изоляция за adapter-интерфейсом (Protocol)

Каждый внешний контракт прячется за Python `Protocol`/adapter в слое
`kb-support-clients`. Доменный код зависит от **доменных** моделей (наши DTO),
не от провизорных форм upstream. Адаптер мапит провизорный JSON → доменный DTO.
Смена upstream-контракта = правка только адаптера + этого ADR.

### Решение 2. Провизорный контракт rehome.one platform API

Базовый URL — из конфига (`KBS_PLATFORM_API_BASE_URL`), без хардкода. Авторизация —
m2m Bearer (Keycloak Client Credentials, `aud: rehome-platform` или согласованный
audience). Все ответы — read-only, кешируются (E3-2, AT-003) с TTL из конфига.

Эндпоинты (провизорно):

```
GET /api/v1/users/{user_id}
  200 -> { "id": uuid, "display_name": str, "email": str|null,
           "phone": str|null, "role": "tenant"|"landlord"|"agent"|"staff",
           "is_active": bool, "created_at": date-time }
  404 -> not found (anti-enumeration на нашей стороне не нужен — m2m)

GET /api/v1/premises/{premises_id}
  200 -> { "id": uuid, "address": str, "kind": str, "rooms": int|null,
           "area_m2": number|null, "landlord_id": uuid|null }

GET /api/v1/bookings/{booking_id}
  200 -> { "id": uuid, "premises_id": uuid, "tenant_id": uuid,
           "landlord_id": uuid, "status": str,
           "period_start": date, "period_end": date|null,
           "monthly_rent": number|null }

GET /api/v1/collaborators/{collaborator_id}
  200 -> { "id": uuid, "name": str, "category": str,
           "contact": { "email": str|null, "phone": str|null }|null,
           "is_active": bool }
```

Семантика ошибок/деградации: 5xx/timeout/circuit-open → клиент возвращает
«контекст недоступен» (None/пусто), НЕ роняет вызывающий код (AT-003). Кеш —
по id с TTL; **не кешировать** для критичных операций (claims/decision), там
данные запрашиваются свежими.

> ПДн в этих ответах (email/phone/имя) — обрабатываются по ФЗ-152: не логируем
> содержимое (mask_pii), храним только в кеше с TTL в своём контуре РФ.

### Решение 3. Провизорный контракт возврата ответа в kb-search

Базовый URL — `KBS_KB_SEARCH_API_BASE_URL` (конфиг). Авторизация — m2m Bearer.

```
POST /api/v1/chat/sessions/{chat_session_id}/operator-reply
  body: { "ticket_id": uuid, "message_id": uuid, "body": str,
          "author": "operator", "sent_at": date-time }
  202 -> принято к доставке в сессию
  404 -> сессия не найдена/истекла (наша реакция: лог + метрика, без падения)
  409 -> сессия закрыта (деградация: лог + метрика)
```

**Инвариант (NFR-1.3, жёстко на нашей стороне в E3-4):** в чат уходят только
сообщения с `is_internal=false` от автора-оператора по заявке с `channel=AI_CHAT`
и непустым `chat_session_id`. Внутренние заметки (`is_internal=true`) **никогда**
не передаются — обязателен security-тест. Идемпотентность доставки — по
`message_id` (повтор не плодит дубль в сессии).

### Решение 4. Версионирование и правило обновления

Провизорность фиксируется в коде ссылкой на этот ADR (`# provisional contract,
see ADR-0006`). При получении реального контракта upstream: обновить ADR (новая
версия/замена), переписать адаптеры, доменный слой не трогать. Контрактные тесты
клиентов идут против **мока** этого контракта (как Prism/фикстуры), а не против
живых соседей.

## Альтернативы

1. **Ждать фиксации upstream-контракта** — отклонено: блокирует ~половину E3 на
   неопределённый срок (зависит от чужих команд), а разблокированный срез
   (from-chat ingest #69, AT-003 фундамент #70) и так готов двигаться.
2. **Прямой доступ к БД платформы / shared-код** — отклонено: нарушает
   архитектурную константу (ADR-0005, отдельный сервис) — не обсуждается.
3. **Строить клиенты без adapter, на форме upstream напрямую** — отклонено: при
   смене провизорного контракта изменения потекут в доменный код по всему слою.
4. **Генерировать клиент из OpenAPI соседей** — отклонено сейчас: у соседей нет
   зафиксированной спеки (в этом и блокер); вернёмся к идее, когда появится.

## Последствия

### Положительные

- E3 разблокирован полностью, можно вести #71/#72 параллельно с #69/#70.
- Чёткая граница: смена upstream = правка адаптера + ADR, без ряби в домене.
- Резилентность (AT-003) и кеш проектируются один раз в E3-2 и переиспользуются.

### Отрицательные / компромиссы

- Риск переделки адаптеров при расхождении провизорного и реального контракта
  (ограничен слоем адаптеров — осознанная цена).
- Контрактные тесты против мока не гарантируют совместимость с живым соседом —
  нужен интеграционный прогон при фиксации upstream (отдельная задача).

### Технические следствия

- Слой `kb-support-clients`: Protocol + adapter на каждый контракт; httpx async,
  resilience из E3-2 (#70).
- Конфиг: `KBS_PLATFORM_API_BASE_URL`, `KBS_KB_SEARCH_API_BASE_URL`, TTL/таймауты —
  переменные окружения, без хардкода.
- Мок-контракт для тестов (фикстуры/Prism-стиль) — артефакт E3-3/E3-4.
- Метки `# provisional contract, see ADR-0006` в коде клиентов.
