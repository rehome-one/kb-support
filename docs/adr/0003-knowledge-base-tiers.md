# ADR-0003: Двухконтурность данных и контроль доступа

## Статус

- [x] **Принято**
- **Дата:** 2026-05
- **Автор:** Архитектор проекта
- **Согласовано Архитектором:** да

## Контекст

База знаний reHome содержит контент с разной чувствительностью:
- Публичные FAQ для гостей сайта
- Карточки квартир с кодами парадных и Wi-Fi паролями (только для нанимателей)
- Внутренние регламенты для сотрудников
- Кадровые документы (HR-restricted)
- Юридические заключения (LEGAL)
- Полная карточка пользователя (только админ)

Требование ФЗ-152: персональные данные доступны только субъектам и сотрудникам
по матрице ролей. Любая утечка через AI-чат, поиск или API — нарушение закона.

Типичная ошибка реализации: фильтрация прав на уровне приложения (после
получения данных из БД). Один баг в коде → утечка ПДн в ответе AI.

## Решение

Принят принцип **двухконтурной архитектуры с фильтрацией на уровне хранилища**.

### Контуры данных

Каждый ресурс (статья, документ, карточка, чанк для RAG) имеет атрибут
`access_level`:

- `PUBLIC` — доступно всем, включая гостей сайта
- `LOGGED` — только для авторизованных пользователей платформы
- `AGENT` — для агентов по закреплённым объектам
- `STAFF` — для сотрудников reHome (поддержка, операторы)
- `LEGAL` — для юристов и руководителей
- `HR_RESTRICTED` — для HR (кадровые данные, не доступно даже админам)

### scope пользователя

Каждый запрос имеет вычисленный `scope`:
- `guest` → PUBLIC
- `tenant` / `landlord` → PUBLIC + LOGGED + свои данные
- `agent` → PUBLIC + LOGGED + AGENT (для закреплённых объектов)
- `staff_support` → PUBLIC + LOGGED + STAFF
- `staff_legal` → PUBLIC + LOGGED + STAFF + LEGAL
- `staff_hr` → PUBLIC + LOGGED + STAFF + HR_RESTRICTED
- `staff_admin` → все уровни, кроме HR_RESTRICTED

### Технические правила (обязательные к исполнению)

1. **`scope` вычисляется ТОЛЬКО бэкендом из проверенного JWT.** Никогда не
   принимается от клиента. Любая попытка передать `scope` в payload запроса
   игнорируется.

2. **Фильтр `access_level` применяется на уровне хранилища, не приложения:**
   - PostgreSQL: `WHERE access_level IN (...)` в SQL-запросе
   - Qdrant: `payload.access_level` filter в search-запросе
   - MinIO: signed URL с TTL только после проверки прав

3. **Никаких `SELECT *` или `find({})` без `access_level` фильтра.**
   Прямые запросы к таблицам с user-generated content без фильтра запрещены.

4. **404 вместо 403** — когда пользователь запрашивает ресурс, который существует,
   но недоступен по правам, возвращается 404. Это защита от утечки информации
   о существовании ресурса.

5. **Маскировка ПДн перед передачей в LLM:**
   Полные паспортные данные, банковские реквизиты, КЭП-сертификаты
   маскируются перед включением в контекст LLM.

6. **Двухконтурный чат:**
   Публичный чат (`scope ∈ {guest, tenant, landlord, agent}`) физически не
   имеет доступа к индексу STAFF+. Внутренний чат (`scope ∈ {staff_*}`) имеет
   доступ к расширенному индексу.

### Тестирование

- **Автотест в CI**: проверка, что нет SQL-запросов к таблицам с
  user-generated content без `WHERE access_level`.
- **Security-тесты**: для каждого endpoint — попытка обхода прав
  (`guest` пытается прочитать `STAFF`, `tenant` пытается прочитать чужие данные).
  Все попытки должны быть отклонены.
- **Контрактные тесты**: проверка маскировки ПДн в ответах API.

## Альтернативы

1. **Фильтрация прав в Python-коде после получения данных** — отклонено:
   один баг в коде → утечка. Не масштабируется на 50+ моделей и эндпоинтов.

2. **Один общий контур с фильтрацией только по `user_id`** — отклонено:
   не покрывает кейс «гость на сайте видит публичный help-центр» и
   «сотрудник видит внутренние регламенты». Слишком грубая модель.

3. **Сложная RBAC-модель с разрешениями на уровне записи** (например, Casbin,
   Open Policy Agent) — отклонено: избыточная сложность для текущего этапа.
   Можно перейти позже, если потребуется.

## Последствия

### Положительные

- Утечка ПДн через ответ API/чата практически невозможна — фильтр в хранилище
- Простая ментальная модель: 6 уровней доступа, понятные scope
- Тестируется автоматически (autotest в CI на отсутствие SELECT без WHERE)
- Соответствие ФЗ-152 — техническая, а не процессная гарантия

### Отрицательные / компромиссы

- Каждая таблица с user-content получает обязательное поле `access_level`
- Migration существующих данных: для всех записей нужно установить уровень
- Сериализаторы в API нужно делать по whitelist полей (зависят от scope)
- Все запросы к Qdrant/Postgres должны иметь явный filter — увеличивает
  объём кода
- Сложность отладки: запросы возвращают «пусто», нужно явно проверять scope

### Технические следствия

- В моделях данных (`Article`, `Document`, `PremisesCard`, `Chunk` и т.д.)
  обязательное поле `access_level`
- В каждом view/endpoint: вычисление scope из request.user → передача в
  query layer как обязательный параметр
- В Django ORM: создать `ScopedQuerySet` mixin для автоматической фильтрации
- В FastAPI: middleware на проверку scope из JWT
- В Qdrant collections: payload field `access_level` индексируется
- В RAG-pipeline: индексация чанков сохраняет `access_level` родительского ресурса
