## Про reference-копии

ADR-0001 / 0003 / 0004 — это **снимки** на момент bootstrap'а 2026-05-30.
Они описывают решения, принятые при разработке kb-platform, которые
переиспользуются в kb-support **по принципу единого технологического и
двухконтурного стиля**.

Если в kb-platform эти ADR будут эволюционировать, **обновлять reference-копии
в этом репо не обязательно**. kb-support — независимый модуль; в нём могут
быть собственные ADR, перекрывающие части reference-снимков. Любое такое
перекрытие — новый ADR с явным указанием «supersedes ADR-NNNN в части XYZ».

ADR-0002 (financial model сервисного платежа kb-platform) **намеренно
пропущен** — не применим к kb-support (модуль не считает деньги сам, см.
ADR-0005 §8.1 ТЗ).

## Workflow

1. **Назрело архитектурное решение** (затрагивает ≥ 2 модуля внутри
   kb-support, или меняет публичный API, или влияет на безопасность).
2. **Создать Issue** с шаблоном `adr-request.md` и описанием контекста +
   альтернатив.
3. **Обсудить** в Issue + чате Claude Code.
4. **Архитектор утверждает** решение и форму ADR.
5. **Создать ADR** копированием `0000-template.md` с новым номером,
   запушить в PR.
6. **Reviewer** ревьюит, что в ADR указаны контекст, альтернативы,
   следствия, тесты соответствия.
7. **Merge** — ADR принят. Дальнейшие изменения — новый ADR с
   `supersedes ADR-NNNN`.

## Связь с ТЗ kb-support v2.2

§8 ТЗ перечисляет 5 принятых решений, образующих ADR-0005. §8.1 фиксирует
**3 открытых вопроса** для E10 (компенсации):

1. Окно подачи 14 дней: жёсткий отказ vs flag `late_submission`.
2. Плата за рассрочку 0.2 %/день: платёжный контур vs kb-support.
3. Scope «claims-оператор»: отдельный vs тег + команда.

Их решение требует отдельных ADR (например, «Обработка просрочки 14-дневного
окна» — следующий свободный номер), которые будут созданы перед стартом E10.
(Номер 0006 занят контрактами E3, см. таблицу выше.)
