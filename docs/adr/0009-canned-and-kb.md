# ADR-0009: Шаблоны ответов и интеграция с базой знаний (для E6)

## Статус

- **Автор:** Агент-Разработчик; принято Архитектором (Evgeniy)

> Этот ADR фиксирует архитектурные решения этапа **E6 «Шаблоны и база знаний»**
> (эпик #19) до старта реализации под-задач #125–#131. Решения приняты Архитектором
> 2026-06-06.

## Контекст

Этап **E6** вводит шаблоны ответов (`CannedResponse`, §3.6 ТЗ v2.2) и интеграцию с
**уже готовой** базой знаний (kb-wiki/kb-search) в режиме read-only: FR-5.1 (CRUD
шаблонов), FR-5.2 (подстановка переменных), FR-5.3 (связь шаблона со статьёй kb-wiki),
FR-5.4 (предложение статей по содержанию заявки через kb-search).

Ключевые ограничения и факты на момент старта:

- **Контракт уже содержит** `CannedResponse`/`CannedResponseInput` и операции
  `listCannedResponses`/`createCannedResponse`/`updateCannedResponse`
  (`docs/openapi.yaml`). Поля: `title`/`body`/`type`(nullable TicketType)/
  `linked_article_slug`(nullable)/`usage_count`. Нет: `getCannedResponse` (by-id),
  render-эндпоинта, suggested-articles. `createMessage` уже принимает
  `canned_response_id` (заглушка, учёт usage_count отложен в E6 — `schemas.py:338`).
- **Инвариант (§4.5 ТЗ, ADR-0005 Решение 1, арх-константа):** kb-wiki/kb-search —
  **только по сети, read-only**. kb-support **ссылается** на статьи и **предлагает**
  их, но **не создаёт и не редактирует** контент БЗ. Без shared-таблиц/кода.
- **Готовые seam'ы:** resilient HTTP-стек AT-003 (#70, `clients/base.py`:
  timeout→breaker→retry→метрики + cache-aside на GET); platform-клиент `HttpPlatformClient` (#71) — источник
  `requester_name` (UserProfile); `clients/kb_search/` (#72) — пока только
  `send_operator_reply`; `clients/auth.py` `TokenProvider`/`StaticTokenProvider`;
  admin-scope `staff_admin` (#86, `auth/scopes.py`).
- **Инфраструктурные блокеры:** реальный m2m-токен для исходящих вызовов (kb-wiki/
  kb-search/platform) — **#77** (m2m-realm не провижинен, ops). До него боевой путь
  интеграций инертен (config-gated), как в E3/E4.
- **ПДн (ФЗ-152):** `requester_name` — персональные данные из platform. Рендер с ПДн
  обязан происходить на сервере; ПДн не должны утекать в клиентский бандл/логи.

Нужно зафиксировать: **чем** подставлять переменные, **какие** переменные доступны,
**где** рендерить, **как** интегрироваться с БЗ (контракт/таймы) и **под каким scope**
живёт CRUD шаблонов.

## Решения

### Решение 1. Шаблонизатор — своя минимальная `{{var}}`-подстановка по белому списку

Подстановка переменных (FR-5.2) — **самописная**: замена токенов `{{var}}` ТОЛЬКО по
**белому списку** известных переменных; никакой логики, выражений, циклов, доступа к
атрибутам. **jinja2 (даже sandboxed) отвергнут**: для простой подстановки избыточен,
добавляет зависимость и поверхность SSTI (шаблоны авторские staff_support, но
рендерятся с ПДн — минимизируем риск). Принцип «разрабатываем сами» (CLAUDE.md).

- Неизвестный/недоступный токен — **оставляется как есть** (документированное
  поведение; не падение, не пустая строка вслепую) — оператор видит, что переменная не
  подставилась, и правит вручную.
- Реализация — чистая функция (`canned/render.py`), unit-тестируемая без I/O.

Отвергнуто: jinja2 SandboxedEnvironment (избыточность + зависимость + SSTI-поверхность);
str.format (ломается на `{`/`}` в тексте, KeyError-хрупкость).

### Решение 2. Набор переменных + рендер на бэкенде

- **Рендер — на бэкенде** (render-эндпоинт `POST /canned-responses/{id}/render` с
  `ticket_id`): единый источник, ПДн остаются на сервере, фронт получает готовый текст
  и вставляет в композер. Фронт **не** подставляет ПДн.
- **Локальные переменные** (из своей БД, доступны сразу): `{{ticket_number}}`,
  `{{ticket_subject}}`, `{{ticket_type}}`, `{{current_date}}` (расширяемо).
- **Платформенные переменные** (`{{requester_name}}` и пр.) — через
  `HttpPlatformClient` (#71), **config-gated по пустому `platform_api_token`** (как
  chat_return/#72 и requester-context/#81): пока #77 не поднят — подставляется пустая
  строка/плейсхолдер, без 5xx. Контракт render-эндпоинта полный сразу; боевое значение
  ПДн включится с #77.

Отвергнуто: рендер на фронте (ПДн в браузер — нарушение ФЗ-152); подстановка только при
отправке сообщения (оператор правит текст до отправки → рендер нужен ДО вставки).

### Решение 3. БЗ-интеграции (kb-wiki + kb-search) — строим сейчас, config-gated

Клиенты kb-wiki (FR-5.3) и kb-search-suggest (FR-5.4) строятся в E6 поверх resilient-
стека AT-003 (#70), по **провизорному контракту** (стиль ADR-0006; провизорный контракт
только в адаптере, метки `# provisional contract`), **config-gated** по пустому
токену/URL (как platform/#71): пока не сконфигурировано — инертны (slug принимается без
валидации; suggested-articles → пустой список), без 5xx. Боевой путь — после ops (#77).

- **kb-wiki** (`clients/kb_wiki/`): проверка существования статьи по `slug`. Валидация
  `linked_article_slug` при create/update CannedResponse: несуществующий slug → 422;
  недоступность kb-wiki → деградация (не блокировать сохранение). Read-only.
- **kb-search**: метод `suggest_articles(query)` поверх существующего
  `clients/kb_search/` (config-gated по `kb_search_*`); эндпоинт
  `GET /tickets/{id}/suggested-articles` (operator-only, 403 заявителю, 404 anti-enum):
  subject/description заявки → kb-search → список статей. Деградация → пустой список +
  флаг, не 5xx.

Отвергнуто: отложить интеграции (FR-5.3/5.4 не закрыты в E6); строить без config-gate
(падали бы без ops-инфры — несогласованно с E3/E4).

### Решение 4. Scope — новый `staff_support` для CRUD шаблонов

FR-5.1 предписывает CRUD под `scope=staff_support+`. Вводим именованный
`STAFF_SUPPORT_SCOPE = "staff_support"` (`auth/scopes.py` + `Principal.is_staff_support`,
как `staff_admin` #86): **CRUD** шаблонов (POST/PATCH) — под `staff_support`;
**list/get/render** — **любой оператор** (нужны для вставки шаблона, FR-2.5). Скоуп
считается бэкендом из проверенного JWT (CLAUDE.md). `staff_admin` остаётся для
конфигурации SLA/automation (§6); шаблоны — рабочий инструмент поддержки, отдельный
скоуп точнее отражает ТЗ.

Отвергнуто: переиспользовать `staff_admin` (отход от буквы ТЗ, смешивает админ-конфиг и
рабочий инструмент); CRUD любым оператором (ТЗ требует ограничения).

### Решение 5. usage_count — best-effort инкремент

При `create_message` с непустым `canned_response_id` — инкремент
`CannedResponse.usage_count` (FR-5.1). **Best-effort:** сбой инкремента/несуществующий
id **не валит** отправку сообщения (лог, не 4xx) — учёт статистики не критичнее самого
ответа. Контракт не меняется (поле уже есть).

## Альтернативы

1. **jinja2 для шаблонов** — отклонено (Решение 1): избыточность, зависимость, SSTI.
2. **Рендер на фронте** — отклонено (Решение 2): ПДн в браузер (ФЗ-152).
3. **Отложить kb-wiki/kb-search** — отклонено (Решение 3): FR-5.3/5.4 не закрыты.
4. **CRUD под staff_admin** — отклонено (Решение 4): отход от ТЗ (staff_support+).
5. **usage_count в транзакции сообщения (fail-together)** — отклонено (Решение 5):
   статистика не должна ронять ответ заявителю.

## Последствия

### Положительные

- E6 **не блокируется** на ops: интеграции config-gated, рендер ПДн config-gated —
  включаются с #77 без правок кода (паттерн E3/E4).
- Безопасность: нет SSTI (Решение 1), ПДн только на сервере (Решение 2), read-only БЗ
  (Решение 3, арх-константа).
- Точный RBAC по ТЗ (Решение 4); статистика не угрожает доставке ответа (Решение 5).

### Отрицательные / компромиссы

- Самописный шаблонизатор ограничен подстановкой (нет условий/циклов) — осознанно
  достаточно для FR-5.2; расширение — при реальной потребности.
- До #77 `requester_name` пуст, suggested-articles пуст, slug не валидируется — как и
  прочие config-gated интеграции (согласованная цена).

### Технические следствия

- Новый пакет `backend/src/api/canned/` (model/schemas/router/repository/render) +
  миграция (revision id ≤32, урок #69); enum-поле `type` — String+валидация (конвенция
  E1, без native PG ENUM). — #125/#126/#127.
- Новый `STAFF_SUPPORT_SCOPE` (`auth/scopes.py` + `Principal.is_staff_support`). — #126.
- Новый под-пакет `clients/kb_wiki/` + метод `suggest_articles` в `clients/kb_search/`
  (провизорный контракт, config-gated, resilient AT-003). — #129/#130.
- Уточнение `docs/openapi.yaml` (getCannedResponse, render, suggested-articles, scope
  аннотации) ⇒ реген `frontend/lib/api/schema.d.ts` (урок #81) + контракт-тест AT-002.
- Best-effort инкремент `usage_count` в `create_message` при `canned_response_id`
  (контракт не меняется — поле уже есть). — #128.
- Render-эндпоинт + фронт-панель шаблонов + suggested-articles (расширение E2). —
  #127/#131.
