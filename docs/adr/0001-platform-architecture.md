# ADR-0001: Базовая архитектура платформы reHome

## Статус

- [x] **Принято**
- **Дата:** 2026-05
- **Автор:** Архитектор проекта
- **Согласовано Архитектором:** да

## Контекст

ООО «РЕХОМ» разрабатывает цифровую платформу долгосрочной аренды жилья.
Модуль базы знаний — отдельный набор приложений, обслуживающих:
- Публичный help-центр
- Внутреннюю wiki для сотрудников
- Хранилище юридических документов
- Менеджер паролей
- Реестр карточек квартир
- Кадровый портал
- AI-чат поверх базы знаний
- Реестр коллаборантов

Ограничения:
- Соответствие ФЗ-152 (все ПДн в РФ, шифрование, аудит, RBAC)
- Бюджетная экономия — не использовать дорогие SaaS-решения
- Долгосрочный контроль над данными — не зависеть от внешних вендоров
- Возможность развития платформы автономно от вендоров

## Решение

Принят принцип **«разрабатываем сами, минимизируем внешние сервисы»**.
Технологический стек разделён на три категории:

### Категория A: Разрабатываем сами

Бизнес-логика, специфичная для reHome:
- `kb-wiki` — внутренняя wiki
- `kb-help` — публичный help-центр
- `kb-files` — хранилище документов
- `kb-vault` — менеджер паролей (security-критичный)
- `kb-staff` — админка реестра квартир и пользователей
- `kb-hr` — кадровый портал
- `kb-search` — RAG-движок для AI-чата

### Категория B: Open-source self-hosted

Готовые инфраструктурные компоненты:
- PostgreSQL 16+ (основная БД)
- pgvector / Qdrant (векторный поиск)
- MinIO (S3-совместимое хранилище)
- Redis (очереди, кеш)
- Dramatiq (фоновые задачи)
- Keycloak (SSO)
- Prometheus + Grafana + Loki (наблюдаемость)

### Категория C: Внешние сервисы (критически необходимый минимум)

Только то, что нельзя сделать самостоятельно по причинам:
лицензирование, специализированный R&D, законодательные требования.

- Банк-партнёр (Точка/Т-банк) — расчётный счёт, эквайринг, номинальный счёт
- Cyclops — KYC по 115-ФЗ
- LLM API (YandexGPT, GigaChat) — генерация ответов AI-чата
- Embeddings API (на старте — внешний, потом self-hosted)
- SMS-провайдер (Exolve) — SMS-OTP
- Контур.Диадок — кадровый ЭДО
- 1С:ЗУП — зарплата и отчётность
- Облачная касса (АТОЛ/Эвотор) — фискальные чеки
- Dadata / Контур.Фокус — справочники, проверка контрагентов
- УЦ Контур — КЭП-токены

## Альтернативы

1. **Полностью SaaS-решения** (Outline + Bitwarden + Helpdeskeddy + Salebot AI) —
   отклонено: vendor lock-in, отсутствие контроля над данными, более высокая
   стоимость владения на горизонте 3 лет, проблемы с глубокой интеграцией с
   платформой rehome.one и AI-чатом.

2. **Только open-source** — отклонено: нерационально разрабатывать собственные
   LLM (стоимость R&D десятки миллионов рублей и команда из 10+ ML-инженеров),
   собственные KYC-системы (требуют банковской лицензии), собственную ККТ
   (требует регистрации с ФНС и поддержки ФФД).

3. **Гибрид с большим количеством SaaS** — отклонено: каждый дополнительный
   внешний сервис увеличивает риск ФЗ-152 (новый процессор ПДн → новый DPA →
   обновление уведомления в РКН) и снижает гибкость.

## Последствия

### Положительные

- Полный контроль над кодом и данными
- Возможность глубокой интеграции с платформой rehome.one
- Меньше расходов на лицензии SaaS (~5-10× ниже стоимость владения)
- Гибкость в выборе LLM-провайдера через абстракцию LLMProvider
- Соответствие ФЗ-152 одной интеграцией с банком, а не каждым SaaS
- Возможность вынести модуль на собственные мощности при росте

### Отрицательные / компромиссы

- Высокая стоимость первоначальной разработки (~7-10 млн ₽)
- Срок до полной готовности: 6-12 месяцев
- Требуется команда минимум 4-6 человек (backend, frontend, DevOps, контент)
- Ответственность за поддержку и безопасность собственного стека
- Высокая планка инженерной дисциплины при разработке kb-vault

### Технические следствия

- Все компоненты размещаются в РФ (требование ФЗ-152)
- Единый SSO через Keycloak: wiki.rehome.one, staff.rehome.one,
  hr.rehome.one, vault.rehome.one, files.rehome.one
- Общая инфраструктура: одна БД на модуль, единый Redis, единый Keycloak
- Категория C проверяется раз в полгода: возможно ли заменить на самописное
