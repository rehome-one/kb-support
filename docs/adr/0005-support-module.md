# ADR-0005: Модуль службы поддержки (kb-support)

## Статус

- [x] **Принято**
- **Дата:** 2026-05
- **Автор:** Архитектор проекта
- **Согласовано Архитектором:** да

## Контекст

Платформе reHome нужна служба поддержки пользователей (нанимателей,
собственников, агентов). Сравнение готовых helpdesk-сервисов (Okdesk,
HelpDeskEddy, Юздеск) показало, что они закрывают потребность, но создают
зависимость от внешнего обработчика ПДн и не интегрируются нативно с уже
построенной архитектурой reHome.

Принято решение разрабатывать собственный модуль `kb-support` (см. ТЗ «Модуль
службы поддержки reHome»), воспроизводящий ядро функционала Okdesk
(заявки, SLA, маршрутизация, каналы, шаблоны), но адаптированный под
предметную область аренды и без избыточного скоупа (выездной сервис, ТОиР,
складской учёт — не реализуются).

Реализация модуля требует фиксации пяти ключевых архитектурных решений,
которые влияют на несколько модулей (`kb-search`, `kb-staff`, модуль
коллаборантов, основная платформа rehome.one) и трудно меняются после начала
кодирования. Этот ADR их фиксирует.

## Решение

### Решение 1. Интеграция с платформой — через API

`kb-support` получает данные о пользователях (User), договорах (Booking),
квартирах (Premises) и коллаборантах (Collaborator) **через внутренний API
основной платформы rehome.one**, а не через прямой доступ к общей БД.

Обоснование: слабая связанность модулей. `kb-support` и основная платформа
могут деплоиться, версионироваться и развиваться независимо. Схема БД одного
модуля не становится жёстким контрактом для другого. Цена — небольшая
дополнительная задержка на сетевые вызовы, которая компенсируется кешированием
часто запрашиваемых справочных данных.

### Решение 2. Граница с ServiceOrder — разные сущности по ссылке

Обращение в поддержку (`Ticket`) и заказ услуги у коллаборанта (`ServiceOrder`
из ADR-0004) — это **две разные сущности, связанные по ссылке**, а не одна,
перетекающая в другую.

- `Ticket` — канал общения с жильцом: «протёк кран», переписка, статусы
  обработки оператором, оценка качества поддержки.
- `ServiceOrder` — фактическая работа коллаборанта (сантехника): свой
  жизненный цикл (принят/в работе/выполнен), эскроу-оплата, выплата исполнителю.

Заявка типа MAINTENANCE может породить ServiceOrder (оператор инициирует заказ
коллаборанту), и тогда в Ticket сохраняется ссылка `service_order_id`. Но это
остаются разные объекты с разными статусами, ответственными и финансовой
логикой.

Обоснование: у них действительно разная природа. Статус «оператор ответил
жильцу» и статус «сантехник выполнил работу» — независимы. Деньги (эскроу)
живут только в ServiceOrder. Смешивание в одну сущность создало бы путаницу
в статусах и нарушило бы границу между поддержкой и маркетплейсом услуг.

### Решение 3. Email-канал — свой IMAP-парсер в контуре РФ

Приём писем на support@rehome.one реализуется **собственным IMAP-парсером,
размещённым в контуре РФ**, без использования внешнего сервиса приёма почты.

Обоснование: ФЗ-152. Письма содержат ПДн пользователей. Внешний почтовый
сервис стал бы дополнительным обработчиком ПДн, что требует DPA и расширяет
периметр риска. Свой парсер держит данные в контуре РФ. Соответствует принципу
ADR-0001 «минимум внешних сервисов».

### Решение 4. Уведомления заявителю — email + push + AI-чат + SMS

На старте заявитель уведомляется о новых ответах оператора и смене статуса
по **четырём каналам**:

- **AI-чат** — если обращение пришло из чата (channel=AI_CHAT), ответ
  возвращается в ту же чат-сессию.
- **Email** — всегда, как базовый канал.
- **Push** — в мобильном приложении / PWA.
- **SMS** — через существующий Exolve (ADR-0001), для критичных уведомлений.

Обоснование: уведомлять стоит там, откуда пришло обращение, плюс
продублировать. SMS включён, поскольку Exolve уже в стеке (используется для
SMS-OTP), дополнительная интеграция минимальна. Логику выбора каналов и
защиту от спама (не слать все 4 канала на каждое мелкое изменение) определить
на этапе реализации.

### Решение 5. UI оператора — минимум на старте

Первая версия рабочего места оператора — **минимальная: список заявок с
фильтрами + карточка заявки** с перепиской, контекстом и действиями.
Полноценное рабочее место с дашбордами, расширенной аналитикой и массовыми
операциями — следующая фаза, по мере роста команды.

Обоснование: на старте 1-3 оператора (зафиксировано). Список с фильтрами
(статус, тип, приоритет, SLA-дедлайн) + карточка закрывают ~90% повседневной
работы. Дашборды и сложная аналитика при трёх операторах избыточны — это
преждевременная оптимизация. Расширение UI оправдано, когда команда вырастет.

## Альтернативы

**По решению 1 (интеграция):**
- Общая БД с внешними ключами — отклонено: жёстко связывает схемы модулей,
  усложняет независимый деплой, нарушает изоляцию.
- Гибрид (критичное через API, справочники через БД) — отклонено как
  промежуточный вариант, усложняющий модель без явной выгоды на старте.

**По решению 2 (ServiceOrder):**
- Заявка MAINTENANCE = ServiceOrder (одна сущность) — отклонено: смешивает
  разные жизненные циклы (общение с жильцом vs работа исполнителя с эскроу),
  создаёт путаницу в статусах и финансовой логике.

**По решению 3 (email):**
- Внешний сервис приёма почты — отклонено по ФЗ-152: лишний обработчик ПДн.

**По решению 5 (UI):**
- Полноценное рабочее место сразу — отклонено: преждевременная оптимизация для
  команды из 1-3 операторов, удлиняет срок до запуска.

## Последствия

### Положительные

- Модуль слабо связан с основной платформой — независимое развитие и деплой.
- Чёткая граница между поддержкой (Ticket) и услугами коллаборантов
  (ServiceOrder) — нет путаницы в статусах и деньгах.
- Все ПДн в контуре РФ — соответствие ФЗ-152 без дополнительных DPA.
- Уведомления по нескольким каналам — выше шанс, что пользователь увидит ответ.
- Минимальный UI — быстрый запуск, фокус на сути.

### Отрицательные / компромиссы

- Интеграция через API добавляет сетевые вызовы — нужно кеширование справочных
  данных (User, Premises) для производительности.
- Свой IMAP-парсер — дополнительная разработка и поддержка (надёжность приёма,
  ретраи, парсинг разных почтовых клиентов).
- Четыре канала уведомлений — нужна логика дедупликации и защиты от спама,
  чтобы не заваливать пользователя.
- Минимальный UI означает, что при росте команды потребуется доработка
  (осознанный технический долг, запланированный).

### Технические следствия

- Слой доступа к данным платформы в `kb-support` — через HTTP-клиент к API
  rehome.one, с кешированием в Redis. Не прямые SQL-запросы к чужим таблицам.
- В модели Ticket — поле `service_order_id` (nullable) для связи с заказом
  коллаборанта. Создание ServiceOrder — вызов API модуля коллаборантов.
- IMAP-парсер — фоновый воркер на Dramatiq, периодический опрос ящика,
  идемпотентное создание/дополнение заявок, привязка по номеру в теме письма.
- Сервис уведомлений — единая точка с роутингом по каналам (AI-чат через
  kb-search, email через свой SMTP, push, SMS через Exolve), с правилами
  дедупликации.
- UI оператора встраивается в `kb-staff` как раздел; на старте — две основные
  страницы (список, карточка).
- Все решения соблюдают ADR-0003 (access_level на уровне хранилища, scope из
  JWT, внутренние заметки недоступны заявителю).
