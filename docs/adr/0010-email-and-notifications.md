# ADR-0010: Каналы email/форма и уведомления заявителю (для E7)

## Статус

> Этот ADR фиксирует архитектурные решения этапа **E7 «Каналы email/форма»**
> (эпик #20) до старта реализации под-задач #142–#151. Решения приняты
> Архитектором 2026-06-06.

## Контекст

Этап **E7** вводит дополнительные каналы приёма заявок и доставку уведомлений
заявителю:

- **FR-1.2** — создание/дополнение заявки из входящего email на support@rehome.one;
  ответы по email привязываются к заявке по номеру в теме.
- **FR-1.3** — создание заявки через веб-форму в личном кабинете.
- **FR-2.3** — ответ заявителю через тот же канал, откуда пришла заявка.
- **FR-6.5** — уведомления заявителю (email, push, в чат) о новых ответах и
  смене статуса.
- **ADR-0005 Решение 3** — email-канал реализуется **собственным IMAP-парсером
  в контуре РФ** (ФЗ-152, без внешнего обработчика ПДн).
- **ADR-0005 Решение 4** — уведомления по **четырём каналам**: AI-чат, email,
  push, SMS (Exolve).

Ключевые ограничения и факты на момент старта:

- **Контракт уже содержит** `TicketChannel.EMAIL`/`WEB_FORM` (`tickets/enums.py`),
  `TicketMessage.attachments: list[str]` (ID файлов в kb-files), `channel`,
  `author_type`, `is_internal`. Готовый паттерн входящего m2m-эндпоинта
  `POST /tickets/from-chat` (#69, только kind=SERVICE) и fire-after-response
  возврата `chat_return` (#72). Идемпотентность from-chat — частичный uniq-index
  по `chat_session_id` (урок: alembic revision id ≤32 символов).
- **Чего ещё нет:** клиента `clients/kb_files/` (загрузка вложений в MinIO по
  API), исходящего SMTP, IMAP-воркера, диспетчера уведомлений.
- **Готовые seam'ы:** resilient HTTP-стек AT-003 (#70, `clients/base.py`:
  timeout→breaker→retry→метрики + cache-aside); `HttpPlatformClient` (#71);
  `clients/auth.py` `TokenProvider`/`StaticTokenProvider`; Dramatiq config-gated
  по `sla_worker_broker_url` (#90/#110, StubBroker → actor инертен); реестр
  системных акторов `auth/system_actors.py` (#106); `chat_return` (#72).
- **Инфраструктурные блокеры (ops):** реальный m2m-токен исходящих вызовов —
  **#77** (m2m-realm не провижинен); Dramatiq broker/worker — **#79** (не поднят);
  IMAP/SMTP-creds и Exolve-creds — ops. До них боевой путь инертен (config-gated),
  как в E3/E4/E5/E6.
- **ПДн (ФЗ-152):** письма содержат ПДн (адреса, тела). Обработка — только в
  контуре РФ; mask_pii перед логированием; вложения-ПДн — в kb-files/MinIO.
- **Анти-спуфинг:** заголовок `From` входящего письма тривиально подделывается;
  доверять ему как источнику личности заявителя нельзя.

Нужно зафиксировать: **как** принимать и отправлять email (инфраструктура,
config-gate), **как** доверять отправителю, **под какой аутентификацией** живёт
веб-форма, **чем** грузить вложения и **как** устроены уведомления (FR-6.5).

## Решения

### Решение 1. IMAP-приём и SMTP-отправка — своя реализация в РФ, config-gated

Приём (IMAP) и отправка (SMTP) email реализуются **самописно в контуре РФ**
(ADR-0005 Решение 3, ФЗ-152, без внешнего почтового сервиса как обработчика ПДн;
принцип ADR-0001 «минимум внешних сервисов»). В backend — только `imaplib`/
`smtplib`/`email` из стандартной библиотеки (без новых внешних сервисов).

- **Приём** — Dramatiq-actor `poll_inbox`, **config-gated по пустому IMAP-конфигу**
  (StubBroker → actor инертен, как SLA-actor #90 и time_based #110). Источник
  истины приёма — IMAP-mailbox; восстановление — повторный poll, не из памяти
  (NFR-3.2). Периодический запуск (cron enqueue) — ops при поднятии воркера, без
  новой зависимости на планировщик.
- **Отправка** — **fire-after-response** через FastAPI BackgroundTasks (паттерн
  возврата в чат #72), **config-gated по пустому SMTP-конфигу**. Durable-доставка
  через Dramatiq — **follow-up** (стиль #79, общий блокер воркера), не в приёмке E7.
- **Идемпотентность приёма** — по `Message-ID` письма (дедуп повторно
  выбранных/доставленных писем), частичный uniq-механизм (как `chat_session_id`
  #69; alembic revision id ≤32).

Отвергнуто: внешний почтовый сервис/SaaS-инбокс (ADR-0005 — лишний обработчик
ПДн, ФЗ-152); хранение очереди приёма в памяти (NFR-3.2 — не переживает рестарт);
синхронная блокирующая отправка SMTP в request-цикле (тормозит ответ оператора).

### Решение 2. Веб-форма — только аутентифицированный заявитель (ЛК)

FR-1.3 — «через веб-форму **в личном кабинете**». Веб-форма принимается
эндпоинтом `POST /tickets/from-web-form` **только от аутентифицированного
заявителя** (`Principal` kind=USER, токен ЛК rehome.one через общий Keycloak):
`requester_id` берётся **из проверенного принципала**, не из тела запроса;
`channel=WEB_FORM`. Анонимного приёма нет — это исключает спам-периметр и
неопознанные ПДн на входе.

- **UI веб-формы живёт в ЛК rehome.one**, вне репозитория kb-support
  (арх-константа: операторский фронт `frontend/` — это kb-staff, не публичный
  ЛК). E7 поставляет **только API + контракт** (OpenAPI + контракт-тест AT-002);
  фронт-компонента в `frontend/` для веб-формы нет.

Отвергнуто: анонимный публичный POST (placeholder в #20) — расширяет периметр
спама/ПДн, требует капчи/anti-abuse, противоречит букве FR-1.3 «в личном
кабинете»; компонент веб-формы в `frontend/` — нарушил бы границу kb-staff
(операторский кабинет) vs публичный ЛК.

### Решение 3. Входящий email — отправитель не доверяется (анти-спуфинг)

Заголовок `From` входящего письма **не является доверенным источником личности**
заявителя (тривиально подделывается).

- Адрес `From` сохраняется в `custom_fields` заявки (трассируемость), **не**
  становится автоматически доверенным requester.
- Резолв `requester_id` — попытка найти `User` по email через
  `HttpPlatformClient` (#71), **config-gated по пустому `platform_api_token`**
  (как requester-context #81): пока #77 не поднят — резолва нет.
- Не нашли / резолв выключен — **sentinel «неподтверждённый email-заявитель»**
  из реестра `auth/system_actors.py` (паттерн `AUTOMATION_ACTOR_ID` #106;
  версионный nibble исключает коллизию с v4); оператор привязывает реального
  заявителя вручную на карточке.
- **Привязка ответов:** номер заявки из subject → активная заявка ⇒ добавить
  `TicketMessage` (requester берётся из заявки, не из письма); номера нет / не
  найдена / заявка CLOSED ⇒ создать новую заявку `channel=EMAIL`.
- **NFR-1.3:** входящее письмо НЕ может выставить `is_internal=true` (внутренние
  заметки — только операторский путь) — security-тест. Spoofed sender → requester
  не доверяется — security-тест.

Отвергнуто: доверять `From` как requester (спуфинг → подмена личности, утечка
ПДн чужой заявки); блокировать письма без распознанного отправителя (теряем
легитимные обращения — лучше создать заявку с sentinel и дать оператору привязать).

### Решение 4. Вложения — новый resilient-клиент kb-files, config-gated

Вложения писем и веб-формы хранятся в **kb-files (MinIO) по API**
(арх-константа: не shared bucket, только сетевой API). Вводится под-пакет
`clients/kb_files/` поверх resilient-стека AT-003 (#70): `KbFilesClient`
Protocol + `HttpKbFilesClient` adapter, **провизорный контракт** (метки
`# provisional contract`, как platform/#71/ADR-0006), **config-gated по пустому
токену/URL** (инертен до #77). Загрузка → `file_id` (UUID) для
`TicketMessage.attachments`. Лимит размера вложения — из Settings (`KBS_*`),
oversized → отклонение (не 5xx); malformed → graceful. Деградация kb-files
(AT-003) → типизированная ошибка, решает вызывающий. Вложения-ПДн не логируются.

Отвергнуто: прямой доступ к bucket MinIO kb-files (нарушение арх-константы —
shared storage); хранение вложений в своей БД (дублирование хранилища, ПДн вне
выделенного контура файлов).

### Решение 5. Уведомления FR-6.5 — диспетчер с дедупом, каналы config-gated

Уведомления заявителю (FR-6.5, ADR-0005 Решение 4) реализуются **диспетчером**
`notifications/dispatch(event)` на событиях (новый публичный ответ оператора;
смена статуса) с **fan-out по четырём каналам** и **дедупом** (не заваливать
заявителя; маркер доставки):

- **AI-чат** — переиспользует `chat_return` (#72) для заявок `channel=AI_CHAT`.
- **Email** — через исходящий SMTP-seam (Решение 1), config-gated.
- **Push** — config-gated seam (инертен до ops).
- **SMS** — через Exolve (уже в стеке per ADR-0001), config-gated seam (creds/
  боевой путь — ops).

Каждый канал — **best-effort, изолированно** (сбой одного не валит прочие и не
роняет заявку; паттерн #72/#107). **NFR-1.3:** `is_internal=true` НЕ
уведомляется ни по одному каналу — security-тест. push/SMS — отдельная под-задача
(#150), ядро+email+чат — #149.

Отвергнуто: рассылка по каждому каналу без дедупа (спам заявителю, прямо названо
риском в ADR-0005); синхронная доставка в request-цикле (тормозит, связывает
доставку с успехом ответа); новый внешний сервис для push/SMS сверх Exolve
(принцип «минимум внешних сервисов»).

## Альтернативы

1. **Внешний почтовый SaaS** — отклонено (Решение 1): лишний обработчик ПДн (ФЗ-152).
2. **Анонимная публичная веб-форма** — отклонено (Решение 2): спам-периметр,
   расходится с FR-1.3 «в личном кабинете».
3. **Доверять `From` как requester** — отклонено (Решение 3): спуфинг → подмена
   личности, утечка ПДн.
4. **Прямой доступ к bucket kb-files** — отклонено (Решение 4): нарушение
   арх-константы (shared storage).
5. **Fan-out уведомлений без дедупа / синхронно** — отклонено (Решение 5): спам
   заявителю, связывание доставки с ответом.

## Последствия

### Положительные

- E7 **не блокируется** на ops: IMAP/SMTP/push/SMS/kb-files/платформенный резолв —
  config-gated, включаются с #77/#79 и ops-creds без правок кода (паттерн E3–E6).
- Безопасность: отправитель email не доверяется (Решение 3), веб-форма
  аутентифицирована (Решение 2), вложения через выделенный контур kb-files
  (Решение 4), is_internal не утекает уведомлением (Решение 5).
- ФЗ-152: вся обработка email в контуре РФ, своя реализация (Решение 1).

### Отрицательные / компромиссы

- Свой IMAP/SMTP — дополнительная разработка и поддержка (надёжность приёма,
  ретраи, разные почтовые клиенты) — осознанная цена ADR-0005.
- До #77 email→requester не резолвится (sentinel + ручная привязка оператором);
  до #79/ops боевой приём/отправка/push/SMS инертны — согласованная цена
  config-gate (как E3–E6).
- Дедуп четырёх каналов уведомлений — нетривиальная логика, чтобы не спамить.

## Тесты соответствия

- **Security:** spoofed sender → requester не доверяется (sentinel); входящее
  письмо не выставляет is_internal; уведомление не уходит по is_internal-заметке.
- **Config-gate:** при пустом IMAP/SMTP/kb-files/Exolve-конфиге — инертно, без
  5xx (как #90/#110/#72/#77).
- **Идемпотентность:** повторный Message-ID не дублирует сообщение/заявку.
- **Контракт:** AT-002 на `from-email`/`from-web-form` + реген schema.d.ts (AT,
  как #81).
- **Арх-константа:** AT-001 (grep запрещённых импортов/SQL) — kb-files только по
  API, без прямого доступа к bucket/чужим БД.
