# ADR-0013: Претензионные типы — case_state, решения, «4 глаза», upstream-seam (для E10)

## Статус

- [x] **Принято**
- **Дата:** 2026-06-08
- **Автор:** Агент-Разработчик; принято Архитектором (Evgeniy)
- **Согласовано Архитектором:** да, 2026-06-08 (открытые вопросы §8.1 Q1-Q3 — ратифицированы 2026-05-30)

> Этот ADR фиксирует архитектурные решения этапа **E10 «Претензионные типы»** (эпик #23)
> до старта реализации под-задач #191–#202. Решения D1-D3 = ратифицированные рекомендации
> §8.1 ТЗ (2026-05-30); D4-D10 ратифицированы 2026-06-08.

## Контекст

E10 реализует претензионные типы `COMPENSATION` / `GUARANTEE` / `INSURANCE` /
`ACCEPTANCE_ACT` (§3.1.1, §3.2.1, §3.3, §3.11, §4.9 FR-9.*, §8.1; Договор найма 5.7/5.8).

**Ключевой факт.** Claims-**контракт уже полный и immutable** (`docs/openapi.yaml` v1.1):
операции `POST /tickets/{id}/decision` и `/case-state`; схемы `TicketCaseState` (8
значений), `TicketDecision` (FULL/PARTIAL/REJECTED), `ActKind` (MOVE_IN/MOVE_OUT),
`SigningStatus` (one_signed/both_signed/disputed), `TicketCaseDetails` (case_type +
typed act_kind/signing_status + payload jsonb); 12 claims-полей на `Ticket`; webhook-события
`ticket.case_decided`/`payout_released`/`insurance_event`. В коде их НЕТ (явные комментарии
«это E10»). **E10 — реализация под существующий контракт, а не его проектирование.**

**Инвариант финмодели (FR-9.8, §8.1, ADR-0005, CLAUDE.md).** kb-support **хранит решение
и ссылки, не считает деньги**. Все денежные операции (выплаты, регресс, плата за рассрочку,
проводки FinancialLedger) — платёжный/юридический контур. Ряд upstream-сущностей ещё не
готов (TD-UP-001).

## Решения

### Решение 1 (D1, §8.1 Q1). Окно подачи 14 дней → флаг `late_submission`

Заявление о компенсации за пределами 14-дневного окна (Договор 5.8.6) **НЕ отклоняется
жёстко на приёме** — принимается и помечается флагом `late_submission`; решение по такой
заявке — за командой legal/finance. Рекомендация §8.1, ратифицирована Архитектором 2026-05-30.

### Решение 2 (D2, §8.1 Q2). Плата за рассрочку 0.2%/день → платёжный контур

Пеня по рассрочке (Договор 5.7.5) **считается в платёжном контуре**, не в kb-support;
тикет лишь отображает/хранит ссылку (при необходимости запрашивает сумму по API).
Рекомендация §8.1, ратифицирована 2026-05-30.

### Решение 3 (D3, §8.1 Q3). claims-оператор → тег + команда, без отдельного scope

Претензионные заявки маркируются **тегом и роутятся на команду legal/finance**; отдельный
RBAC-scope `claims-operator` НЕ вводится (пересмотр возможен при росте объёма). Рекомендация
§8.1, ратифицирована 2026-05-30.

### Решение 4 (D4). Размещение claims-данных — плоские поля на Ticket + TicketCaseDetails (jsonb)

**12 плоских nullable-колонок** на `Ticket` (case_state, claim_amount, approved_amount,
decision, decision_reason, decision_notified_at, payout_due_at, linked_payment_id,
regress_obligation_id, policy_id, insurance_event_id, acceptance_act_id) + сущность
**`TicketCaseDetails` 1:1** (case_type; typed `act_kind`/`signing_status`; `payload` jsonb +
Pydantic-валидатор по case_type).

> **Решение Архитектора (честная пометка).** Контракт (immutable) уже размещает эти 12 полей
> на `Ticket` — следуем контракту. При этом §3.11 ТЗ мотивировал ОБРАТНОЕ — выносить детали в
> отдельную сущность, «чтобы не раздувать Ticket десятками nullable-колонок». Баланс выбран
> Архитектором: индексируемые / SLA-критичные поля — плоские на `Ticket` (по контракту);
> тип-специфичный payload — в `TicketCaseDetails.payload` (jsonb + валидатор, §8.1). Поля
> `act_kind`/`signing_status` — **typed top-level по контракту** (не внутри payload, как в
> табличке §3.11).

### Решение 5 (D5). Машина case_state — ALLOWED_TRANSITIONS (проектируется в E10-2)

Машина состояний `case_state` по образцу `tickets/state_machine.py` (таблица переходов;
запрещённый переход → 422; каждый переход — в TicketHistory §3.7). Терминалы: PAID, REJECTED.

> **Решение Архитектора (честная пометка).** §3.2.1 ТЗ задаёт только **линейную цепочку**
> `CLAIM_SUBMITTED → DOCS_PENDING → UNDER_REVIEW → [INSPECTION] → DECISION_MADE →
> PAYOUT_PENDING → PAID | REJECTED`. Детальная таблица ALLOWED_TRANSITIONS — включая
> достижимость `REJECTED` из `UNDER_REVIEW` и `DECISION_MADE`, опциональность `INSPECTION`,
> терминальность PAID/REJECTED — это **проектное решение сверх буквы ТЗ**, финализируется при
> реализации E10-2 (#192) и фиксируется тестами.

### Решение 6 (D6). «4 глаза» на PAYOUT_PENDING → PAID через AutomationRule

Переход `case_state PAYOUT_PENDING → PAID` требует подтверждения **двумя РАЗНЫМИ
сотрудниками** (§3.2.1, FR-9.4, §8.1, CLAUDE.md): первый аппрув фиксирует `actor_id`, второй
(actor_id ≠ первого) разрешает переход; идемпотентно; реализуется AutomationRule, а не волей
одного оператора. Дублирующая проверка — на стороне `releasePayout` (§8.1). Механика
фиксации первого подтверждающего — проектируется в E10-4 (#194).

### Решение 7 (D7). Upstream — config-gated seam «фиксация решения + ссылки»

Пока upstream-сущности не готовы (TD-UP-001: `AcceptanceAct`, `PaymentReleaseChecker`,
`FinancialLedger`, `BankProvider.releasePayout`, webhook страховщика), E10 работает в режиме
**«фиксация решения + ссылки»** (FR-9.8): хранятся UUID-ссылки (`linked_payment_id`,
`regress_obligation_id`, `acceptance_act_id`, `insurance_event_id`), **деньги не считаются**.
Клиенты к upstream — **config-gated seam'ы** (resilient AT-003; инертны до ops, как #77/#79).
Связь только по сети/id, **без FK к чужим БД** (арх-константа). Боевой путь включается
провижинингом.

### Решение 8 (D8). Маршрутизация по типу/каналу через AutomationRule on_create

`COMPENSATION` (канал LK_CLAIM) → команда **legal+finance**; `GUARANTEE` (SYSTEM) →
**finance**; `INSURANCE` (INSURER_WEBHOOK) → **support**; `ACCEPTANCE_ACT` → **support**.
Реализуется правилами AutomationRule (E5), не хардкодом.

### Решение 9 (D9). Каскад ACCEPTANCE_ACT MOVE_OUT + ущерб → связанный COMPENSATION

При выезде (`act_kind=MOVE_OUT`) с зафиксированным ущербом — **системно создаётся связанный
COMPENSATION-тикет** (линк через `case_details`/ссылку).

> **Решение Архитектора (честная пометка).** FR-9.7 описывает фиксацию расхождений акта и
> создание связанного COMPENSATION при ущербе на выезде в общем виде; конкретный механизм
> каскада (кто инициирует, как линкуется) — проектное решение, детализируется в E10-9 (#199).

### Решение 10 (D10). Приём COMPENSATION — валидация комплекта

Валидация комплекта вложений (5.8.6); окно 14 дней → флаг `late_submission` (D1); сумма
ущерба **> 50 000 ₽ → флаг требования отчёта независимого оценщика** (FR-9.1, §3.3.1);
доказательства-вложения — в kb-files/MinIO по `access_level` (ФЗ-152, #143).

## Альтернативы (сводно)

- **Жёсткий отказ на 14-дневном окне** — отвергнуто (D1): лишает legal/finance возможности
  решить по существу; §8.1 рекомендует флаг.
- **Все claims-детали в jsonb (без плоских колонок)** — отвергнуто (D4): контракт уже
  размещает индексируемые/SLA-поля на Ticket; jsonb-only усложнил бы фильтрацию/SLA.
- **Все claims-детали плоскими колонками на Ticket** — отвергнуто (D4): §3.11 справедливо
  предостерегает от раздувания Ticket; тип-специфичное — в payload.
- **Отдельный RBAC-scope claims-operator** — отвергнуто (D3): тег+команда достаточно на старте.
- **Реальные выплаты сейчас** — невозможно (D7): upstream не готов (TD-UP-001).

## Последствия

### Положительные

- Полный claims-путь на стороне kb-support (модель, case_state, decision, «4 глаза»,
  маршрутизация, приём, SLA, webhook) под уже зафиксированный контракт.
- Все денежные/upstream-зависимости — config-gated seam'ы, инертны до ops; боевой путь
  включается провижинингом (как остальной E7/E8).

### Отрицательные / компромиссы

- До провижининга upstream выплаты/регресс/проводки — режим «фиксация + ссылки» (no-op
  деньги). Принято (FR-9.8).
- Двойное размещение (плоские поля + jsonb) требует дисциплины валидатора payload по типу.

### Технические следствия

- E10-1 (#191): enums + миграция 12 полей на Ticket + TicketCaseDetails 1:1 + payload-валидатор.
- E10-2 (#192): машина case_state (финализация таблицы D5) + `/case-state`.
- E10-3 (#193): `/decision` (legal/finance gate, reason обяз. PARTIAL/REJECTED).
- E10-4 (#194): «4 глаза» (механика D6).
- E10-5 (#195): приём каналов + валидация COMPENSATION (D10) + маршрутизация (D8).
- E10-6 (#196): SLA по Договору. E10-7 (#197): upstream seam'ы (D7). E10-8 (#198): webhook.
- E10-9 (#199): ACCEPTANCE_ACT + каскад (D9). E10-10 (#200): GUARANTEE/INSURANCE.
- E10-11 (#201): фронт. E10-12 (#202): security-тесты.

## Тесты соответствия

- Запрещённый переход case_state → 422; каждый переход — в TicketHistory.
- `/decision` без команды legal/finance → 403; reason обязателен при PARTIAL/REJECTED.
- «4 глаза»: PAYOUT_PENDING→PAID одним и тем же actor_id невозможно (нужен второй, ≠ первого).
- COMPENSATION > 50 000 ₽ → флаг отчёта оценщика; вне 14 дней → `late_submission`.
- Upstream config-gated: без провижининга — инертно (ссылки хранятся, деньги не считаются),
  не ошибка; деградация AT-003.
- ФЗ-152: доказательства в kb-files по access_level; claims-ПДн не в логах.
- Деньги kb-support не считает (FR-9.8) — суммы/проводки только хранятся/отображаются.
