# ADR-0011: Аналитика — панель супервайзера, отчёты и метрики (для E8)

## Статус

- [x] **Принято**
- **Дата:** 2026-06-08
- **Автор:** Агент-Разработчик; принято Архитектором (Evgeniy)
- **Согласовано Архитектором:** да, 2026-06-08

> Этот ADR фиксирует архитектурные решения этапа **E8 «Аналитика»** (эпик #21)
> до старта реализации под-задач #165–#172. Решения приняты Архитектором

## Контекст

Этап **E8** вводит аналитику службы поддержки:

- **FR-7.1** — панель супервайзера: открытые заявки, нарушения SLA, загрузка
  операторов, среднее время ответа/решения.
- **FR-7.2** — отчёты: объём заявок по типам/каналам/периодам, соблюдение SLA,
  удовлетворённость (рейтинги), повторные обращения, эффективность операторов.
- **FR-7.3** — метрики для дашборда Grafana: rate заявок, containment rate
  AI-чата (доля диалогов без эскалации), время в очереди.
- **§2.1 (роли)** — **Супервайзер = `staff_support` + `manage`** (распределение,
  контроль SLA, отчёты, управление операторами); Администратор = `staff_admin`;
  Оператор = `staff_support`.

Ключевые ограничения и факты на момент старта:

- **Контракт уже содержит** операции `GET /api/v1/support/stats`
  (`getSupportStats` → схема `SupportStats`: секции `period`/`tickets`/`sla`/
  `performance`/`quality`/`ai_chat`) и `GET /api/v1/support/reports/{type}`
  (`getReport`, `type ∈ {volume, sla, satisfaction, reopens, operators}`,
  `format ∈ {json, csv}`). Но `stats` и `reports` слабо специфицированы:
  у обеих операций только ответ `403`, без аннотации scope и `401`; тело отчёта —
  `additionalProperties: true` (нетипизировано, как `SLAPolicy`/`AutomationRule`
  до E4/E5). `period.from`/`period.to` — `format: date`.
- **Данные — в своей БД:** `Ticket` (type/channel/status/priority/team,
  `created_at`/`resolved_at`/`first_responded_at`, `sla_*_due_at`,
  `sla_paused_seconds`, `rating`/`rating_comment`, `reopened_count`,
  `assignee_id`/`requester_id`); `TicketHistory` (неизменяемый аудит, 5 лет,
  NFR-1.4). Истинный containment AI-чата (знаменатель = всего AI-сессий) живёт
  в **kb-search**, у нас — только эскалированная сторона (заявки `channel=AI_CHAT`).
- **Чего ещё нет:** пакета `backend/src/api/analytics/`; scope супервайзера;
  типизированных схем отчётов; CSV-экспорта; frontend-страниц дашборда/отчётов;
  Grafana-дашбордов.
- **Готовые seam'ы:** resilient HTTP-стек AT-003 (#70, `clients/base.py`:
  timeout→breaker→retry→метрики + cache-aside) и кэш (`Cache` Protocol +
  `InMemoryCache`/`RedisCache` в `clients/cache.py`); клиент kb-search (#72/#130,
  `clients/kb_search/`); `clients/auth.py` `TokenProvider`/`StaticTokenProvider`;
  именованные scope'ы `auth/scopes.py` + флаги `Principal` (#86 staff_admin /
  #126 staff_support); единый breach-предикат `tickets/sla_query.py` (#90);
  SLA-метрики `tickets/sla_metrics.py` (#91, `sla_breaches_total`, TTFR/TTR);
  паттерн типизации OpenAPI + контракт-тест AT-002 (#86/#104); изоляция тестов на
  общей тест-БД (#111).
- **Инфраструктурные блокеры (ops):** реальный m2m-токен исходящих вызовов —
  **#77** (m2m-realm не провижинен); Grafana-провижининг — ops. До них боевой
  путь containment инертен (config-gated), как в E3–E7.
- **ПДн (ФЗ-152):** отчёты не должны выгружать сырые ПДн; метрики Prometheus —
  низкой кардинальности без ПДн.
- **Арх-константа (NFR-4.4):** kb-support — отдельный сервис, своя БД; аналитика
  считается **только по своей БД**, доступ к чужим данным — только по API.

Нужно зафиксировать: **под какой авторизацией** живёт аналитика, **как** и **где**
агрегируются метрики, **как** считается containment AI-чата (которого у нас нет
целиком), и **в каком часовом поясе / за какой период** агрегируются данные.

## Решения

### Решение 1. RBAC аналитики — новый именованный scope `staff_supervisor`

Доступ к `GET /support/stats` и `GET /support/reports/{type}` требует **нового
scope `staff_supervisor`**. Оператор (только `staff_support`) → **403**.

- **Перевод формулировки ТЗ.** ТЗ §2.1 описывает роль Супервайзера как
  `staff_support + manage` — где «manage» это *капабилити*, а не отдельный scope.
  Фиксируем «manage» как **именованный scope `staff_supervisor`** (а не голую
  строку-капабилити `manage`), следуя паттерну именованных скоупов без магических
  строк (#86 ввёл `staff_admin`, #126 — `staff_support`): `auth/scopes.py`
  `STAFF_SUPERVISOR_SCOPE` + `Principal.is_staff_supervisor`. Это сознательный
  технический перевод буквы ТЗ, а не расхождение.
- **Семантика включения — аддитивная (решение Архитектора 2026-06-08).** Роль
  «Супервайзер» в Keycloak мапится на **ОБА** scope: `staff_support` И
  `staff_supervisor` (ровно `staff_support + manage` из §2.1). Поэтому супервайзер
  параллельно работает с заявками как оператор (через `staff_support`), а
  аналитику открывает `staff_supervisor`. **Код НЕ выводит один scope из другого**
  (`is_staff_support` не возвращает true для держателя `staff_supervisor` без
  `staff_support`) — scope'ы независимы, оба выдаёт Keycloak по маппингу роли. Это
  держит RBAC простым и явным; #166 (gate stats/reports) и #172 (RBAC-матрица)
  трактуют единообразно: проверка аналитики = строго `is_staff_supervisor`.
- Супервайзер ≠ Администратор: `staff_admin` настраивает конфигурацию (SLA-политики,
  правила, шаблоны), `staff_supervisor` мониторит и распределяет. Это разные роли
  §2.1, поэтому отдельный scope, а не переиспользование `staff_admin`.

Отвергнуто: гейт под `staff_admin` (размывает роль — по §2.1 супервайзер ≠ админ);
гейт под `staff_support` (любой оператор видел бы отчёты эффективности операторов —
функция супервайзера); голая строка-капабилити `manage` (магическая строка вне
паттерна именованных скоупов).

### Решение 2. Агрегация — SQL on-the-fly по своей БД + Redis cache-aside (TTL)

Сводные метрики считаются **SQL-агрегатами (GROUP BY) по своей БД на запрос**, с
кэшированием результата по схеме **cache-aside с коротким TTL**.

- Переиспользуем существующий кэш-слой #70: `Cache` Protocol +
  `InMemoryCache`/`RedisCache` (`clients/cache.py`). Ключ кэша — namespaced по
  типу запроса и нормализованному периоду; инвалидация — **по TTL** (аналитика
  толерантна к небольшой задержке свежести). **Недоступность Redis не валит
  запрос** — деградация на InMemory/прямой расчёт (как cache-aside в адаптерах
  #71), без 5xx.
- **Без внешнего warehouse / OLAP / ETL** — арх-константа (NFR-4.4): аналитика
  только по своей БД kb-support. Индексы под агрегаты (`created_at`, `status`) —
  по мере необходимости в #165 (рукописные миграции, если понадобятся; на
  bootstrap-объёмах full-scan приемлем).

Отвергнуто: материализованные представления / периодический ETL в сводные таблицы
(преждевременная оптимизация на bootstrap-объёмах; усложняет инвалидацию); внешний
аналитический контур / отдельная БД (нарушил бы двухконтурность и арх-константу).

### Решение 3. AI containment rate — config-gated seam к kb-search

`ai_chat.containment_rate_pct` тянется из **kb-search по API** (провизорный
контракт в духе ADR-0006, resilient AT-003, **config-gated** по пустому токену —
как #72/#130).

- Истинный containment (доля диалогов без эскалации) требует знаменателя «всего
  AI-сессий», который живёт в kb-search, не у нас. Поэтому: выключено или
  деградация апстрима → `containment_rate_pct = null` + флаг деградации в секции
  `ai_chat`. **`escalated_count`** (заявки `channel=AI_CHAT` за период) kb-support
  **считает сам всегда** по своей БД.
- Запрос к kb-search передаёт только агрегат-параметры (период), **без ПДн**.
  Боевой путь — после **#77** (m2m-токен); до тех пор секция честно деградирует
  (паттерн config-gate E3–E7).

Отвергнуто: вычислять containment только по своей стороне (нет знаменателя →
метрика была бы неверной); хардкод/заглушка containment (костыль — мок в проде).

### Решение 4. Период и часовой пояс — UTC, дефолт последние 30 дней

Агрегация ведётся в **UTC**: границы суток считаются в UTC, `from`/`to`
**включительно**. При **отсутствии** `from`/`to` бэкенд подставляет окно
**`[today − 30 дней; today]` в UTC** (а не `400`) — это контрактное поведение, не
только реализационная деталь.

- Якоря: volume-метрики — по `created_at`; SLA-compliance и performance — по
  соответствующим меткам (`resolved_at`, `first_responded_at`).
- **Инвариант нулевого знаменателя:** при пустом периоде (нет заявок) или нулевом
  знаменателе rate/avg/compliance возвращаются как **`null`/`0`**, а **не** `5xx`
  и не `NaN`. Это нормативное требование (проверяется в #172).

Отвергнуто: TZ из активного `BusinessHours` (добавляет зависимость расчёта от
конфига и сложность DST на границах суток — избыточно для тех/операционных
отчётов); явный `tz`-параметр в query (расширяет контракт и поверхность
тестирования без явной потребности). Отчёты E8 — операционные, не бухгалтерские;
UTC детерминирован и достаточен. Пересмотр (TZ-aware периоды) возможен позже
отдельным ADR при появлении потребности.

### Доп. решения (фиксируются здесь, отдельного согласования не требуют)

- **CSV-отчёты = агрегаты без сырых ПДн (ФЗ-152).** `operators`-отчёт — это
  `assignee_id` + счётчики/средние; **`rating_comment` НЕ выгружается** (только
  числовой `rating` в агрегатах `satisfaction`); никаких email/телефонов/тел
  сообщений. CSV — RFC4180, stdlib `csv`.
- **Grafana — JSON-файлы в репо** (`docs/grafana/`), провижининг в Grafana —
  **ops** (стиль «код/конфиг в репо, инфра у ops», как #79/#90). #169 коммитит
  JSON-дашборд(ы) + how-to.

## Последствия

### Положительные

- E8 **не блокируется** на ops: containment config-gated, включается с #77 без
  правок кода (паттерн E3–E7). Остальная аналитика работает на своей БД сразу.
- Чёткое разделение ролей §2.1: оператор / супервайзер / админ — разные scope'ы.
- Арх-константа соблюдена: аналитика только по своей БД; kb-search — по API.
- ФЗ-152: отчёты — агрегаты без сырых ПДн; метрики без ПДн.

### Отрицательные / компромиссы

- SQL on-the-fly масштабируется до определённого объёма; при росте потребуется
  материализация/индексы — осознанно отложено (follow-up при росте).
- До #77 containment деградирован (null) — согласованная цена config-gate.
- UTC-границы суток могут не совпадать с «рабочим днём» супервайзера в МСК —
  принято для простоты и детерминизма; пересмотр — отдельным ADR при потребности.

### Технические следствия

- Новый пакет `backend/src/api/analytics/`: `period.py` (чистая модель периода,
  UTC, дефолт 30д, валидация `from<=to`→422); `queries.py`/`repository.py`
  (SQL-агрегаты, переиспользуют breach-предикат `tickets/sla_query.py` #90);
  `service.py` (сборка `SupportStats` + cache-aside #70). — #165.
- `GET /support/stats`: новый scope `STAFF_SUPERVISOR_SCOPE` (`auth/scopes.py`) +
  `Principal.is_staff_supervisor`; уточнение контракта (`401`/`403` + scope-
  аннотация); **расширение схемы `SupportStats.ai_chat` флагом деградации**
  (поле, отражающее отсутствие upstream) + **реген `frontend/lib/api/schema.d.ts`**
  (ожидаемое контрактное изменение, не drive-by — урок #81); config-gated seam
  `clients/kb_search`. — #166.
- `GET /support/reports/{type}`: типизированные read-only схемы 5 отчётов вместо
  `additionalProperties: true` (whitelist полей) + CSV (`text/csv`); контракт-тест
  AT-002 + реген schema.d.ts. — #167.
- Метрики `analytics/metrics.py`: `tickets_created_total{type,channel}`,
  `support_queue_time_seconds` (histogram); **переиспользуют** `sla_breaches_total`
  (#91), не дублируют. Экспорт через существующий `/metrics`. — #168.
- Grafana JSON в `docs/grafana/` + how-to. — #169.
- Frontend: `app/dashboard/` (панель супервайзера) и `app/reports/` (отчёты +
  CSV-скачивание), server-only fetch токеном на сервере, gate супервайзера. —
  #170/#171.
- Новые ключи Settings (`KBS_*`): kb-search analytics URL/token (config-gate
  containment), TTL кэша аналитики — с **пустым/безопасным дефолтом**.

## Тесты соответствия

- **RBAC (security):** оператор (`staff_support`) → `403` на stats и всех 5
  reports; супервайзер (`staff_supervisor`) → `200` (#166/#172).
- **Config-gate:** при пустом kb-search-конфиге containment → `null`/degraded, без
  `5xx`; недоступность Redis → расчёт без кэша, без `5xx` (#165/#166).
- **Период/инвариант:** границы суток UTC включительно; отсутствие `from/to` →
  `[today-30d; today]`; `from>to` → `422`; пустой период → `null`/`0`, не `5xx`/
  `NaN` (#172).
- **ФЗ-152:** CSV/JSON отчётов не содержат `rating_comment` и иных сырых ПДн;
  лейблы метрик без ПДн (#167/#168).
- **Контракт:** AT-002 на `getSupportStats`/`getReport` + реген schema.d.ts (#166/
  #167).
- **Арх-константа:** AT-001 (grep запрещённых импортов/SQL) — агрегация только по
  своей БД, kb-search только по API.
