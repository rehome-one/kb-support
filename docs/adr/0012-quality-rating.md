# ADR-0012: Оценка качества — рейтинги, уведомление о низких, email-CTA (для E9)

## Статус

- [x] **Принято**
- **Дата:** 2026-06-08
- **Автор:** Агент-Разработчик; принято Архитектором (Evgeniy)
- **Согласовано Архитектором:** да, 2026-06-08

> Этот ADR фиксирует архитектурные решения этапа **E9 «Оценка качества»** (эпик #22)
> до старта реализации под-задач #183–#185. Решения приняты Архитектором 2026-06-08.

## Контекст

Этап **E9** вводит оценку качества обслуживания:

- **FR-8.1** — после **закрытия** заявки заявителю предлагается оценить (1-5) и
  оставить комментарий.
- **FR-8.2** — низкие оценки (1-2) автоматически уведомляют супервайзера.

Что уже есть (факт кода, не дублируем):

- `POST /api/v1/support/tickets/{id}/rate` (E1 #12): `actions.rate()` —
  **requester-only** (`router.py`: не-REQUESTER → 403), статусы `_RATEABLE_STATUSES =
  {RESOLVED, CLOSED}`, пишет `Ticket.rating`/`rating_comment` + history `rated`.
  В `actions.rate()` уже стоит seam-комментарий «Низкая оценка (1-2) → уведомление
  супервайзера/метрика — FR-8.2, E9 (#22)».
- Поля `Ticket.rating: int|None`, `Ticket.rating_comment: Text|None`.
- Инфраструктура уведомлений (E7): пакет `notifications/` (dispatcher / channels
  push-SMS-seam #150 / dedup через `custom_fields` / labels), email-outbound
  (`email/outbound.py`, config-gated SMTP #147), best-effort fire-after (#72/#149).
- Аналитика: `analytics` `quality_stats` (avg_rating, ratings_count, #165),
  satisfaction-отчёт (#167), Prometheus-паттерн `tickets_created_total` (#168).

Открытый вопрос — **где живёт UI оценки заявителя** (заявитель в рабочее место
оператора `frontend/` не ходит), **как адресуется «супервайзер»** (в kb-support он
не персонифицирован), **на каком переходе слать CTA**, **как считать низкие оценки**.

## Решения

### Решение 1 (D1). UI оценки заявителя — в ЛК rehome.one (вне репо) + email-CTA

Форму оценки (1-5 + комментарий) показывает **ЛК rehome.one**, вне репозитория
kb-support — тем же принципом, что UI веб-формы (**ADR-0010 Решение 2**, #148 в E7).
ЛК отправляет оценку в существующий `POST /tickets/{id}/rate`. kb-support строит
**email-CTA «оцени заявку»** (Решение 3) со ссылкой на страницу оценки в ЛК.

Рабочее место **оператора** (`frontend/`) оценку **только отображает** (read-only:
рейтинг + комментарий на карточке) — submit заявителя туда не встраивается
(оператор ≠ заявитель; арх-константа: kb-support = только API, requester-UI вне репо).

### Решение 2 (D2). FR-8.2 уведомление супервайзера — config-gated seam

«Супервайзер» в kb-support **не персонифицирован** (нет такого контракта ни в БД,
ни в platform API). Поэтому уведомление о низкой оценке — **config-gated seam**:
адресат/маршрут берётся из `Settings` (например, `low_rating_notify_email` и/или
push-канал), реальная доставка/адрес — зона ops, до провижининга seam инертен (как
notify-каналы #149/#150, #77).

Реализационно **переиспользуем инфраструктуру уведомлений** (`notifications/`,
паттерн #149) — это переиспользование машинерии, **не** прямое следствие ADR-0005
Решение 4 (то — про каналы уведомлений **заявителю**; адресат FR-8.2 — внутренний,
staff). Маршрутизация именно **супервайзеру** — **решение Архитектора** (E9), а не
буква ADR-0005. Срабатывание — **best-effort fire-after**: сбой уведомления НЕ
роняет `rate` (как #72).

### Решение 3 (D3). Email-CTA FR-8.1 — на переходе в CLOSED

CTA «оцени заявку» слать заявителю при переходе заявки в **CLOSED** (буква FR-8.1
«после **закрытия**»). Механика:

- **fire-after** на закрытии (как status-уведомления #149), **config-gated SMTP**
  (#147) — без `smtp_host` инертно;
- ссылка на страницу оценки в ЛК — **URL-шаблон из config** (не хардкод);
- **дедуп** — маркер в `Ticket.custom_fields` реассайном словаря (как #149; in-place
  мутация JSONB не персистится) → один CTA на закрытие;
- слать **только если `rating` ещё `None`** (уже оценил — CTA не нужен).

### Решение 4 (D4). Низкая оценка — порог ≤2; Prometheus-метрика распределения

«Низкая» = `rating <= 2` (FR-8.2 «1-2»), **именованная константа** (не магическое
число). Наблюдаемость — Prometheus **`ticket_ratings_total{rating}`** (счётчик
распределения 1-5, низкая кардинальность, паттерн #168), инкремент на `rate()`.
`quality_stats` (#165) и satisfaction-отчёт (#167) **уже покрывают** avg/распределение
в аналитике — их НЕ дублируем; метрика #168-стиля нужна для Grafana-rate.

### Решение 5 (D5). Eligibility оценки — без изменений

Оставляем как в E1: **requester-only**, статусы **RESOLVED | CLOSED**, повторная
оценка перезаписывает (overwrite-идемпотентно). Не сужаем.

> Осознанная асимметрия с D3: `rate` допускается на RESOLVED **и** CLOSED, а email-CTD
> слать только на CLOSED (буква FR-8.1). Это не баг — заявитель может оценить и на
> RESOLVED (если зашёл в ЛК сам), но проактивно зовём только после закрытия.

### Решение 6 (D6). ФЗ-152 / NFR-1.3 — `rating_comment` как ПДн

`rating_comment` — свободный текст заявителя, потенциально содержит ПДн. Поэтому:

- **не логировать** комментарий (в логах — только id заявки/факт оценки; при
  необходимости маскировать);
- **не выводить в метрики** — `ticket_ratings_total` несёт только число `rating`,
  не текст (согласовано с ADR-0011 «`rating_comment` НЕ выгружается»);
- в **email заявителю** (CTA) комментарий не фигурирует;
- уведомление супервайзера (внутренний контур) может нести комментарий, но без
  утечки в внешние каналы.

## Альтернативы (сводно)

- **UI оценки в `frontend/` kb-support** — отвергнуто: фронт = рабочее место
  оператора, заявитель туда не ходит; потребовало бы requester-сессий в нашем UI и
  нарушило бы роль фронта (ADR-0010 Реш.2).
- **Platform-lookup супервайзера** (m2m по команде заявки) — отвергнуто: #77 blocked
  на ops, и platform не отдаёт «супервайзера команды» (нет контракта). Преждевременно.
- **Маршрут уведомления по команде заявки** — отвергнуто: «супервайзер» ≠ команда;
  адресов команд в kb-support тоже нет — упёрлось бы в тот же config-seam.
- **CTA на RESOLVED и CLOSED** — отвергнуто: FR-8.1 буквально «после закрытия»;
  двойной CTA усложнил бы дедуп.

## Последствия

### Положительные

- Полный путь оценки на стороне kb-support (API есть, добавляем CTA + FR-8.2 + метрика
  + read-only отображение) без нарушения арх-константы.
- Все внешние зависимости (SMTP, адресат супервайзера, ЛК-URL) — config-gated,
  инертны до ops; боевой путь включается провижинингом, как остальной E7.

### Отрицательные / компромиссы

- FR-8.2 до провижининга `Settings`-адресата фактически no-op (seam инертен) — как
  notify-каналы #150 / m2m #77.
- Асимметрия eligibility (D5) vs CTA-trigger (D3) — задокументирована, не баг.

### Технические следствия

- E9-1 (#183): хук в `actions.rate()` seam — `rating<=2` → уведомление супервайзера
  (config-gated seam, recipient из `Settings`, best-effort fire-after) + Prometheus
  `ticket_ratings_total{rating}`.
- E9-2 (#184): email-CTA на CLOSED (fire-after, config-gated SMTP, дедуп custom_fields,
  ЛК-URL из config, только при `rating=None`).
- E9-3 (#185): read-only отображение рейтинга + комментария на карточке оператора.

## Тесты соответствия

- Порог низкой оценки строго `rating <= 2` (2 → уведомление, 3 → нет).
- Дедуп CTA: один CTA на закрытие; не слать при уже выставленном `rating`.
- `rating_comment` НЕ попадает в логи / метрики / email заявителю (security-тест).
- FR-8.2 best-effort: смоделированный сбой уведомления НЕ роняет `rate` (200/коммит).
- Оператор-фронт оценку только отображает (нет submit-контрола; requester-UI вне репо).
- Config-gated: без `Settings`-адресата/`smtp_host` пути инертны (no-op, не ошибка).
